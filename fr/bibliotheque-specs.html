Bibliothèque de SPECs — des gabarits éprouvés

Neuf SPECs de référence pour les domaines techniques récurrents, livrées par npx aiad-sdd init dans .aiad/specs-library/. C'est l'équivalent SDD de l'« Extended Guidance » : un point de départ structuré, jamais un raccourci autour du cycle.

Ce que contient un gabarit

Chaque gabarit est une SPEC au format EARS (Format : EARS déclaré en entête) avec un frontmatter pré-rempli : domaine, statut draft, champs title et parent_intent à substituer, et surtout la gouvernance Tier 1 pré-câblée — les agents réglementaires concernés sont déjà déclarés dans le champ governance. À la Gate, le linter ears-validator passe donc d'office en mode strict.

Un gabarit n'est pas une SPEC validée

Le statut est draft par construction : un gabarit n'a ni Intent parent, ni auteur humain, ni contexte projet. Il ne franchit jamais l'Execution Gate tel quel.

Le câblage de gouvernance suit la logique des déclencheurs Tier 1 : tout gabarit manipulant des données utilisateur embarque AIAD-RGPD ; rag-llm et multi-tenant-saas (scoring d'usage) déclenchent AIAD-AI-ACT ; notifications-multi-channel produit de l'interface, donc AIAD-RGAA ; les domaines gourmands en ressources (observabilité, recherche, inférence) embarquent AIAD-RGESN.

Quand partir d'un gabarit

  • Domaine régulé ou contractuel — paiement, export RGPD, composant IA : le format EARS strict et la gouvernance pré-câblée évitent les angles morts coûteux.
  • Pattern d'architecture connu — multi-tenant, OIDC, OTel : le gabarit encode les cas limites que l'on découvre d'habitude en production.
  • Montée en compétence — un gabarit est aussi un exemple de SPEC bien structurée pour calibrer ses propres rédactions.

Comment adapter un gabarit

  1. Capturer l'intention d'abord : /sdd intent — le gabarit ne remplace ni le POURQUOI ni l'auteur humain ; renseignez parent_intent.
  2. Vérifier la viabilité : /sdd research ancre le gabarit dans votre code réel — un gabarit générique peut être inadapté à votre architecture (verdict DEFER ou NO-GO possible).
  3. Adapter la SPEC : copier le gabarit dans .aiad/specs/, substituer les champs, restreindre le périmètre à une tâche atomique (/sdd split si besoin).
  4. Repasser la Gate : /sdd gate — le SQS s'applique au gabarit adapté comme à toute SPEC ; 0 violation EARS donne le bonus sur le critère Testabilité.
Anti-pattern

Lancer /sdd exec directement sur un gabarit brut. Sans Intent parent ni adaptation au contexte, c'est précisément le code générique que le SDD Mode existe pour éviter — et la Gate le bloquera de toute façon (pas d'auteur humain identifiable).
