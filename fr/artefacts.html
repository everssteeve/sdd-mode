Hiérarchie documentaire

Les artefacts — la mémoire collective de l'équipe

Les artefacts AIAD ne sont pas de la bureaucratie : ce sont des outils de pensée et de communication. Ils rendent visible ce qui resterait implicite, alignent l'équipe sur une compréhension partagée, et fournissent aux agents IA le contexte dont ils ont besoin pour performer. Sans eux, le contexte se perd à chaque session.

Outils de pensée, pas paperasse

Un bon artefact se reconnaît à quatre qualités : il est actionnable (peut-on agir à partir de ce document ?), vivant (évolue-t-il avec la compréhension ?), minimal (contient-il juste assez, pas plus ?) et collaboratif (a-t-il été co-créé, pas dicté ?). L'important n'est pas d'avoir des documents parfaits — c'est d'avoir des documents utiles. Sans AGENT-GUIDE, les agents produisent du code générique ; sans ARCHITECTURE, du code incohérent ; sans SPEC, du code incorrect.

La hiérarchie documentaire

Les artefacts s'organisent en hiérarchie stricte. La règle est absolue : ne jamais contredire un niveau supérieur.

Constitution AIAD (valeurs immuables)
  └── Agents de Gouvernance Tier 1 (droit de veto)
       └── PRD (vision produit)
            └── ARCHITECTURE (standards techniques)
                 └── AGENT-GUIDE (contexte permanent)
                      └── SPEC (activation par tâche)

Cette hiérarchie est aussi l'outil de gestion du Context Engineering Budget : le contexte stratégique en haut (injecté en cadrage), le contexte permanent au milieu (condensé), le contexte de tâche en bas (une SPEC à la fois). En cas de conflit entre une SPEC et un agent de gouvernance, la gouvernance prévaut — toujours.

Intent Statement — le POURQUOI archivé

L'Intent Statement est l'artefact de premier ordre : il capture l'intention humaine derrière chaque fonctionnalité en cinq champs (POURQUOI MAINTENANT, POUR QUI, OBJECTIF, CONTRAINTES, CRITÈRE DE DRIFT), avec un auteur humain obligatoire — principe Human Authorship. Il précède et transcende la SPEC, et il est archivé dans .aiad/intents/. Dans six mois, c'est lui qui répondra à la question « qu'est-ce qu'on voulait accomplir, et pourquoi ? ».

PRD — la valeur attendue

Le Product Requirement Document clarifie pourquoi et quoi construire avant le comment. Il contient le contexte et le problème, les Outcome Criteria mesurables, les personas et use cases, le hors périmètre volontaire, les trade-offs et les risques. Cinq pratiques sont non négociables : partir du problème (pas de la solution), définir les critères de succès avant de construire, le rédiger avec l'équipe, le maintenir vivant, et l'illustrer (wireframes, flows).

Anti-pattern

Le PRD fleuve de cinquante pages que personne ne lit, ou le PRD vague type « Améliorer l'expérience utilisateur ». Les deux sont inutiles. Et côté agents : le PRD complet n'est injecté qu'en phase de cadrage, jamais en développement.

ARCHITECTURE — le référentiel de cohérence

Le document ARCHITECTURE définit les standards techniques que les agents et les PE doivent respecter, en neuf rubriques : principes architecturaux, vue d'ensemble, stack technique justifiée, structure du projet, conventions de code, patterns et bonnes pratiques, sécurité, performance, et ADR (Architecture Decision Records). Il est évolutif, justifié, pragmatique (YAGNI), visuel et actionnable. En session agent, il n'est jamais injecté en entier : un condensé permanent de 500 tokens maximum suffit — le reste se consulte à la demande.

AGENT-GUIDE — le contexte permanent de l'agent

L'AGENT-GUIDE est le document le plus opérationnel du framework : injecté dans chaque session, il transforme un agent générique en agent du projet. Il contient l'identité du projet, les règles absolues TOUJOURS / JAMAIS (ex. : toujours synchroniser SPEC + code dans la même PR ; jamais pusher de secrets), les conventions de code, le vocabulaire métier, les patterns et anti-patterns — et deux sections d'apprentissage complémentaires :

Lessons Learned vs Human Learnings
SectionQuestion poséeMise à jour
Lessons Learned« Qu'est-ce que l'agent a mal fait ? » — erreurs récurrentes de l'agent et corrections appliquéesFin de chaque itération (5 min, pas optionnel)
Human Learnings« Qu'est-ce que l'humain a mal exprimé ? » — défaillances de l'expression de l'intentionRevue à l'Atelier d'Intention mensuel

Un guide de cent règles est moins efficace qu'un guide de dix règles bien choisies : l'AGENT-GUIDE est borné par le Context Budget (condensé ≤ 500 tokens en session).

SPECs — l'instruction de travail de l'agent

Une SPEC fait le pont entre l'intention (PRD, Intent) et l'implémentation par l'agent. Une SPEC de qualité permet de générer 80 %+ du code correct du premier coup — c'est l'investissement au meilleur ROI du framework. Quatre critères la définissent : atomicité, précision, testabilité, complétude — scorés par le SQS à l'Execution Gate (≥ 4/5 requis). Chaque SPEC porte son propre Critère de Drift et reste synchronisée avec le code à chaque PR.

Les Definitions of Done

La Definition of Output Done (DoOD) établit le standard pour qu'un incrément soit livrable, en six catégories : techniques, sécurité, performance, fonctionnels, déploiement, review. Le principe est non négociable : tous les critères ou rien — « Done à 90 % n'est pas Done ».

La Definition of Outcome Done (DoOuD) mesure si la valeur attendue a été réalisée : User Outcomes (NPS, adoption, rétention), Business Outcomes (MRR, conversions, efficacité) et Learning Outcomes (hypothèses validées ou invalidées, insights). L'output est le moyen ; l'outcome est le but — une fonctionnalité livrée mais non adoptée n'est pas un succès.

Anti-patterns des artefacts

Trois façons de tuer ses artefacts

Le guide de cent pages — Context Budget dépassé, l'agent ignore la plupart des règles. La SPEC figée — « validée, on ne la touche plus » : en cinq itérations, c'est le spec drift et l'agent travaille sur une base erronée. Tout injecter en même temps — l'agent reçoit tout, comprend peu : c'est le context rot. AGENT-GUIDE pour le permanent, SPEC pour la tâche — ne jamais mélanger les deux.
