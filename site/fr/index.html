Framework AIAD v1.18.0 — Open Source

Le développement piloté par la spec,
pour l'ère des agents IA

SDD Mode transforme une intention humaine en code vérifié : Intent Statement, SPEC vivante, Execution Gate, Drift Lock et gouvernance réglementaire — sans jamais déléguer la paternité de l'intention.

Le cœur du framework

Un cycle, sept étapes, zéro code sans spec validée

Le développement suit un cycle strict — on ne saute jamais d'étape. Chaque étape produit un artefact traçable, du POURQUOI humain jusqu'au verrou anti-drift de la PR.

  1. 1. Intent Statement

    L'intention humaine en 5 champs : pourquoi maintenant, pour qui, objectif, contraintes, critère de drift. Auteur humain obligatoire.

    Lire l'article →

  2. 2. Research

    Discovery ancrée dans le code, verdict gradué GO / CONDITIONAL GO / DEFER / NO-GO — tranché par l'humain.

    Lire l'article →

  3. 3. SPEC

    La spécification vivante : scope, interface, comportement, cas limites, tests. Format prose ou EARS strict.

    Lire l'article →

  4. 4. Execution Gate

    Le SQS score la SPEC sur 5 critères + le Test de l'Étranger. Gate fermée sous 4/5 — aucun agent ne démarre.

    Lire l'article →

  5. 5. Exécution agent

    Contexte optimisé (AGENT-GUIDE + ARCHITECTURE condensée + une seule SPEC), plan d'implémentation avant le code.

    Lire l'article →

  6. 6. Validation

    Trois axes : technique, fonctionnel, gouvernance. Les 4 agents Tier 1 disposent d'un droit de veto fail-closed.

    Lire l'article →

  7. 7. Drift Lock

    Code et SPEC synchronisés dans la même PR. Le drift est un échec de processus, pas une erreur d'agent.

    Lire l'article →

Fondations

Trois principes fondateurs

  • Spec as Living Invariant

    La SPEC n'est pas un document de passage : elle reste la source de vérité entre l'intention humaine et le code agent — avant, pendant et après l'implémentation. AIAD est spec-anchored : ni spec-first jetable, ni spec-as-source.

    La SPEC vivante →

  • Drift = échec de processus

    Du code qui évolue sans ses artefacts n'est pas terminé. La mise à jour de la SPEC fait partie de la Definition of Done — vérifiée algorithmiquement par la matrice de traçabilité.

    Le Drift Lock →

  • Context Engineering Budget

    Le Product Engineer est responsable du budget de contexte de chaque session : 60-70 % de la fenêtre au maximum, une seule SPEC à la fois, Intent en tête de contexte.

    Le Context Budget →

Tier 1 — droit de veto

La gouvernance réglementaire intégrée au cycle

Quatre agents de gouvernance sont injectés dans chaque session qui touche leur périmètre. Verdict UNKNOWN = VETO : la politique est fail-closed, jamais d'auto-blanchiment silencieux.

  • AIAD-AI-ACT

    Règlement (UE) 2024/1689. Classification de risque et obligations dès qu'un composant IA est impliqué — ML, LLM, scoring, recommandation.

    En savoir plus →

  • AIAD-RGPD

    Privacy by Design : minimisation, bases légales, droits des personnes — dès que le code traite des données personnelles.

    En savoir plus →

  • AIAD-RGAA

    Accessibilité numérique RGAA 4.1 / WCAG 2.1 — dès que le code produit une interface utilisateur.

    En savoir plus →

  • AIAD-RGESN

    Écoconception RGESN v2 : sobriété computationnelle sur toute décision technique — performance, ressources, dépendances.

    En savoir plus →

L'écosystème humain

Cinq responsabilités, pas de rôles rigides

AIAD ne définit pas des postes mais des responsabilités à assumer — une personne peut en porter plusieurs. La règle d'or : chaque responsabilité a un porteur clairement identifié.

  • Product Manager

    Responsable de la valeur. Construit-on la bonne chose ?

    Détails →

  • Product Engineer

    Gardien de l'intention. Orchestre les agents IA sans trahir le POURQUOI.

    Détails →

  • Agents Engineer

    Responsable de l'écosystème IA et du Harness Engineering — permissions minimales.

    Détails →

  • QA Engineer

    Qualité intégrée dès le départ — les tests exploratoires restent 100 % humains.

    Détails →

Constitution AIAD

Sept valeurs immuables

Elles ne sont pas des aspirations — elles sont constitutives du framework. En retirer une fait s'effondrer l'ensemble.

  1. 1. Primauté de l'Intention Humaine

    L'agent exécute. Il ne décide pas de ce qui mérite d'être construit.

  2. 2. Honnêteté sur les Contradictions

    Dire la vérité sur les trade-offs plutôt que promettre l'impossible.

  3. 3. Sobriété Intentionnelle

    Construire le nécessaire, pas l'impressionnant.

  4. 4. Ouverture Radicale

    Open source : le savoir méthodologique se partage, il ne se verrouille pas.

  5. 5. Empirisme sans Concession

    Valider par l'observation, pas par le consensus.

  6. 6. Responsabilité Partagée

    Succès et échec appartiennent à l'équipe — jamais de culpabilisation.

  7. 7. Human Authorship

    La paternité de l'intention ne se délègue pas. Si l'agent a rédigé l'Intent, la session est invalidée.
