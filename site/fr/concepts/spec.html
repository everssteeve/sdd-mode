Étape 3 du cycle · /sdd spec

SPEC — la spécification vivante

La SPEC fait le pont entre l'intention métier et l'implémentation concrète par les agents IA. C'est l'instruction de travail de l'agent pour une fonctionnalité donnée — et l'investissement au meilleur ROI du framework : une SPEC de qualité permet à un agent de générer plus de 80 % du code correct du premier coup.

Spec as Living Invariant

Le premier principe fondateur du SDD Mode positionne la SPEC comme un invariant vivant : elle n'est pas un document de passage, elle reste la source de vérité entre l'intention humaine et le code agent — avant, pendant et après l'implémentation. Trois postures existent dans l'écosystème spec-driven :

Comparaison des trois postures spec-driven
PostureDevenir de la specLimite
Spec-firstRédigée puis abandonnée après la tâcheObsolète dès la deuxième itération
Spec-as-sourceLa spec génère le codeCouplage rigide, le code réel diverge du générateur
Spec-anchored (AIAD)Ancre permanente, synchronisée avec le code à chaque PRExige la discipline du Drift Lock

L'argument n'est pas que documentaire : une session ancrée sur une SPEC consomme en moyenne 41,7 % de tokens en moins qu'une session sans spécification (R2Code, arXiv 2026). La SPEC est un point d'ancrage stable contre le context rot.

Les douze sections d'une SPEC

Une SPEC AIAD est normalisée. Elle contient :

  1. le scope d'activation — injectée seule (par défaut) ou combinée avec d'autres SPECs (fonctionnalités transverses) ;
  2. le contexte — référence à l'Intent Statement parent, objectif, outcome attendu ;
  3. le périmètre — in scope et out of scope explicites ;
  4. les fichiers impactés — à créer ou à modifier, liste exhaustive ;
  5. l'interface technique — endpoints, types, schémas de base de données ;
  6. le comportement détaillé — flow nominal et cas limites (au moins 3 edge cases explicites) ;
  7. les règles de validation — avec schémas ;
  8. les règles métier à appliquer ;
  9. les tests attendus — scénarios à implémenter ;
  10. les exemples d'usage — concrets, requête/réponse ;
  11. la Definition of Done (DoOD) — les critères de « Done » ;
  12. le Critère de Drift — la règle concrète pour détecter que la SPEC est devenue obsolète.
**Critère de Drift** : cette SPEC est en drift si le fichier
`auth/validator.ts` est modifié sans mise à jour de la section
Interface Technique.
Anti-pattern

La SPEC tentaculaire à vingt fonctionnalités, ou la SPEC vague type « Améliorer la performance ». Les deux sont inutilisables — et coûtent des heures de correction du code généré, là où corriger la SPEC aurait pris dix minutes.

Les quatre critères de qualité

Les quatre critères d'une SPEC de qualité
CritèreBon exempleMauvais exemple
AtomicitéUne seule responsabilité fonctionnelle (1 SPEC = 1 PR)Un module entier
Précision« Retourner 400 si title vide ou > 200 caractères »« Gérer les erreurs »
Testabilité« Accepter test@example.com, rejeter invalid »« Tester la validation »
ComplétudeTypes, validation, edge cases et tests inclusLe « happy path » seul

Ces quatre critères alimentent directement le Spec Quality Score évalué à /sdd gate. Une SPEC qui ne les satisfait pas ne doit jamais être soumise à un agent. Au-delà de 200 lignes, la SPEC est trop grande : direction /sdd split.

La variante EARS (v1.11) — l'option anti-ambiguïté

Depuis la v1.11, les critères d'acceptation peuvent être rédigés au format EARS (Easy Approach to Requirements Syntax) via /sdd spec --ears. La SPEC est alors basée sur le template .aiad/specs/spec-ears-template.md et déclare Format : EARS en entête — c'est ce signal qui active le linter strict ears-validator à la Gate. Chaque critère suit l'un des cinq patterns :

Ubiquitous          The <system> SHALL <response>.
Event-driven        WHEN <trigger>, the <system> SHALL <response>.
State-driven        WHILE <state>, the <system> SHALL <response>.
Optional feature    WHERE <feature>, the <system> SHALL <response>.
Unwanted behaviour  IF <condition>, THEN the <system> SHALL <response>.

Le linter applique les règles R1–R7 : mots interdits (should, might, fast, user-friendly, intuitive…), un seul SHALL par critère, déclencheur explicite (WHEN/WHILE/IF/WHERE), sujet explicite, verbe observable (return, log, reject — jamais understand), quantification chiffrée (« within 200ms p95 », pas « rapidement ») et absence de conjonctions composées.

Effet à la Gate

0 violation EARS → bonus garanti de +1 sur le critère SQS 2 (Testabilité). Une seule violation → critère 2 forcé à 0 et Gate fermée, avec plan de remédiation R1–R7 critère par critère.

EARS reste optionnel : les formats prose et EARS cohabitent, aucune migration n'est imposée. À réserver aux SPECs où l'ambiguïté coûterait cher : sécurité, paiement, conformité réglementaire, contrats d'API publics. En contrepartie : environ +30 % de verbosité et une courbe d'apprentissage.

Option REASONS Canvas (SPDD)

Quand l'Intent parent est complexe ou ambigu, /sdd spec propose le REASONS Canvas — l'outil SPDD de Kevlin Henney — comme étape de cadrage : il structure la justification de l'intention avant de basculer au format de SPEC AIAD standard. Les deux approches sont compatibles : le Canvas enrichit le POURQUOI, il ne remplace pas le format de SPEC.

Découper avec /sdd split

Quand la Gate échoue sur l'atomicité, que la SPEC dépasse 200 lignes, que l'agent exige plus de deux relances ou que le budget de contexte explose, on découpe. Trois patterns guident le découpage :

Patterns de découpage d'une SPEC
PatternAxe de découpeExemple
VerticalPar couche technique042a Backend · 042b Frontend · 042c Intégration
FluxPar cas d'usage042a Happy path · 042b Edge cases · 042c Error handling
ContratInterface d'abord, puis dépendances042a Setup/contrat · 042b Logique · 042c Wiring

Chaque sous-SPEC doit pouvoir passer la Gate indépendamment, et la somme des sous-SPECs couvre 100 % de la parente. Quand plusieurs SPECs sortent du découpage, /sdd spec produit un plan de parallélisme (waves et graphe de dépendances) validé par le PE avant la rédaction.

Signal d'alarme

Plus de 5 sous-SPECs après découpage ? L'Intent est trop ambitieux : remonter à /sdd intent plutôt que de spécifier un monstre. Le découpage n'est pas un échec — c'est un signe de maturité du Product Engineer.
