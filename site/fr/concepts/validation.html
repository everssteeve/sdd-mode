Étape 6 du cycle · /sdd validate

Validation — trois axes, un verdict

Le code qui compile n'est pas forcément le code qui fonctionne, et le code qui fonctionne n'est pas forcément le code qui respecte l'intention. La validation SDD est triple — technique, fonctionnelle, gouvernance — et le code n'est jamais terminé tant que la SPEC n'est pas synchronisée (Drift Lock).

Axe 1 — Validation technique

Le socle objectif : la chaîne d'outillage du projet, exécutée intégralement et reportée en tableau. Aucun jugement, des faits.

Checks de la validation technique
CheckCommandeRésultat
Lint[commande projet]PASS / FAIL
Types[type-check]PASS / FAIL
Tests unitaires[test]PASS / FAIL
Tests d'intégration[test:integration]PASS / FAIL
Build[build]PASS / FAIL
Couverture[coverage]X %
Deux interdits absolus

Ne JAMAIS valider un code qui ne passe pas les tests. Ne JAMAIS valider un code contenant des TODO-JNSP: non résolus — ces marqueurs sont des questions en attente de décision humaine, pas des détails cosmétiques.

Axe 2 — Validation fonctionnelle

L'axe fonctionnel confronte le code aux cas de test de la SPEC, critère par critère — contre le code lui-même, pas seulement contre les tests. Entrées spécifiées respectées, étapes de traitement suivies, format de sortie produit, chaque cas limite couvert par un test, tous les critères d'acceptation cochés. En complément, le Critère de Drift de l'Intent parent est relu : le signal observable de trahison de l'intention est-il absent ?

Axe 3 — Validation gouvernance

Les quatre agents Tier 1 — AIAD-AI-ACT, AIAD-RGPD, AIAD-RGAA, AIAD-RGESN — sont appliqués via la skill regulatory-veto. Chacun dispose d'un droit de veto : un VETO bloque, un WARN exige un plan de remédiation avant merge. Cette validation n'est pas optionnelle pour les projets concernés.

UNKNOWN = VETO (fail-closed)

Quand un agent de gouvernance ne peut pas décider — référentiel non applicable de façon certaine, information manquante — il retourne UNKNOWN. Ce verdict est traité comme un VETO : on ne merge pas sur un doute réglementaire. C'est la déclinaison gouvernance de la politique JNSP.

Une composition de skills, sans duplication

/sdd validate est la commande la plus composite du cycle : elle n'implémente aucune logique propre mais compose trois skills réutilisables — ears-validator (couverture des critères testables de §3), drift-detection (écarts code ↔ SPEC sur le diff) et regulatory-veto (les 4 référentiels Tier 1). Chaque skill est aussi invoquée ailleurs dans le cycle (/sdd gate, /sdd drift-check, /sdd security…) : la logique vit en un seul endroit, les commandes l'orchestrent.

Le verdict

Les quatre issues de la validation
VerdictSignificationAction
VALIDÉTriple validation passéeProcéder au Drift Lock (/sdd drift-check)
CORRECTIONSÉcarts mineurs et localisésLister, relancer l'agent sur les points précis
REJETÉchec structurelSPEC imprécise → Human Learning · agent erroné malgré SPEC claire → Lesson Learned
JNSPValidation indécidableQuestion(s) à l'humain — ne pas dégrader en « corrections mineures » pour faire avancer le merge

La distinction Human Learning / Lesson Learned est essentielle pour l'amélioration continue : la première documente un écart entre intention humaine et livraison (la SPEC était le problème), la seconde une erreur récurrente de l'agent (l'exécution était le problème). Les deux alimentent l'AGENT-GUIDE.

Les rôles complémentaires — /sdd security et /sdd audit

Deux commandes d'audit prolongent la validation pour les fonctionnalités à fort enjeu, avec persistance des rapports dans .aiad/metrics/ :

Commandes d'audit complémentaires
CommandePérimètrePersistance
/sdd securityOWASP Top 10, gestion des secrets, permissions des agents (minimal necessary permissions), exposition des données, conformité AI-ACT/RGPD.aiad/metrics/security/YYYY-MM-DD-SPEC-NNN.md
/sdd auditConformité code ↔ SPEC, dette technique introduite (complexité, couplage, lisibilité), cohérence avec l'AGENT-GUIDE.aiad/metrics/audit/YYYY-MM-DD-SPEC-NNN.md
Quand les lancer

/sdd security : après toute fonctionnalité touchant des accès, des données utilisateur, des secrets ou un composant IA — recommandé avant toute PR critique, avec un modèle frontier. /sdd audit : pour les fonctionnalités à fort enjeu technique, ou après plusieurs itérations sur un même composant. Les rapports persistés alimentent les métriques et les rétrospectives.
