Glossaire — le vocabulaire du SDD Mode

Tous les termes du framework, classés par ordre alphabétique — du vocabulaire fondateur (Intent Statement, SPEC, Drift Lock) aux notions introduites par les versions 1.7 à 1.18 (routers, skills, JNSP, EARS, traçabilité machine-vérifiable, Phase Research).

A

AGENT-GUIDE
Contexte permanent de l'agent : règles absolues, conventions du projet, Lessons Learned, Human Learnings et renvois de gouvernance. Injecté condensé (max 500 tokens) dans chaque session. Depuis la v1.12, c'est la source amont des règles multi-runtime émises par emit-rules. Voir les artefacts.
AGENT-GUIDE Gouvernance
Les 4 guides réglementaires Tier 1 dotés d'un droit de veto : AIAD-AI-ACT, AIAD-RGPD, AIAD-RGAA, AIAD-RGESN. Détail sur la page Gouvernance.
Anti-Drift Check
Rituel de fin d'itération vérifiant la synchronisation artefacts / code sur l'ensemble du projet — l'application périodique du Drift Lock. Commande : /sdd drift-check.
ARCHITECTURE
Document des standards techniques : stack, patterns, décisions structurantes. Injecté condensé (~500 tokens) dans chaque session agent — jamais en intégralité. Rédigé via l'assistant /sdd arch.
Atelier d'Intention
Rituel mensuel (60 min max) qui pose la question : « construisons-nous toujours ce que nous voulions ? ». Espace humain pur — les agents IA n'y ont aucun rôle, pas même un résumé automatique. Préparé par /aiad intention, jamais animé par lui. Voir Boucles & rituels.

C

Context Engineering Budget
Capacité d'absorption de contexte d'une session agent, sous la responsabilité du Product Engineer. Règles : une session = un objectif, une seule SPEC active, 60-70 % de la fenêtre comme maximum effectif, Intent et SPEC en tête de contexte. Audité par /sdd context. Voir l'article Context Engineering Budget.
Context Rot
Dégradation de la qualité d'un LLM avant la limite théorique de sa fenêtre de contexte : réponses moins précises, contraintes de SPEC oubliées, comportements répétitifs. Justifie le seuil opérationnel de 60-70 % et la SPEC comme point d'ancrage stable.

D

Discovery
Exploration obligatoire du code réel pendant la Phase Research : la viabilité d'une intention se vérifie dans le dépôt, pas dans une discussion abstraite. Le hook discovery-gate.js vérifie ce prérequis avant /sdd spec et /sdd exec.
DoOD — Definition of Output Done
Critères de complétion d'une tâche, définis dans la SPEC : code, tests, artefacts synchronisés. La mise à jour de la SPEC en fait partie — un code mergé avec une SPEC désynchronisée n'est pas terminé.
DoOuD — Definition of Outcome Done
Pendant orienté valeur de la DoOD : la fonctionnalité n'est réellement « done » que lorsque l'outcome visé par l'Intent est observé chez les utilisateurs — pas seulement lorsque le code est livré.
DORA Metrics
Les 4 indicateurs de performance de livraison du programme DORA : Deployment Frequency, Lead Time for Changes, Change Failure Rate, MTTR. Calculés organiquement par /aiad dora avec positionnement Elite / High / Medium / Low. Voir Métriques — DORA.
Drift Lock
Politique de PR : code et SPEC synchronisés dans le même commit. Étape 7 du cycle, enforced par hook pre-commit et par la matrice de traçabilité en CI. Voir l'article Drift Lock.

E

EARS — Easy Approach to Requirements Syntax
Variante optionnelle (v1.11) de rédaction des SPECs : exigences en SHALL avec déclencheurs WHEN / WHILE / IF / WHERE. Activée par /sdd spec --ears, lintée en mode strict (règles R1-R7) à la Gate : 0 violation → +1 garanti sur le critère SQS Testabilité ; ≥ 1 violation → critère forcé à 0. À réserver aux SPECs où l'ambiguïté coûte cher — sécurité, paiement, conformité, contrats d'API.
Execution Gate
Point de contrôle entre la SPEC et le lancement de l'agent : aucun code avant Gate ouverte, c'est-à-dire SQS ≥ 4/5. Commande : /sdd gate. Voir l'article Execution Gate & SQS.

F

Fact Technique
Artefact de /sdd fact : capture d'un écart constaté entre comportement livré et comportement désiré, avec qualification d'impact (fonctionnel / sécurité / performance / conformité) et décision d'action corrective tracée dans .aiad/facts/FACT-NNN.md.
Fail-closed
Politique d'incertitude de la gouvernance : quand un agent Tier 1 ne peut pas trancher, le verdict UNKNOWN vaut VETO. On bloque par défaut plutôt que de laisser passer — jamais d'auto-blanchiment silencieux. Voir Gouvernance.
Flow Metrics
Les 5 indicateurs de santé du flux de valeur : Cycle Time, Lead Time, Throughput, WIP, Flow Efficiency. Calculés par /aiad flow, avec décomposition du Lead Time pour localiser les goulots. Voir Métriques — Flow.

G

GO / CONDITIONAL GO / DEFER / NO-GO
Verdict gradué de la Phase Research sur la viabilité d'une intention : viable (GO), viable sous conditions explicites (CONDITIONAL GO), à reporter (DEFER), non viable (NO-GO). Le verdict est tranché par l'humain — l'agent instruit le dossier, il ne décide pas. Voir Phase Research.

H

Harness Engineering
Pratique centrale de l'Agents Engineer : configuration, supervision et gouvernance des agents selon le principe des minimal necessary permissions. Auditée par /sdd security.
Human Authorship
Valeur 7 du framework : la paternité de l'intention ne se délègue pas. Tout Intent Statement est rédigé par un humain identifiable — si un agent l'a rédigé, la session est invalidée. Vérifié par la skill human-authorship-check. Voir l'article Human Authorship.
Human Learnings
Section de l'AGENT-GUIDE documentant les écarts entre intention humaine et livraison — alimentée par l'Atelier d'Intention. À distinguer des Lessons Learned, qui documentent les erreurs de l'agent.

I

Intent Statement
Artefact de premier ordre capturant le POURQUOI humain en 5 champs : POURQUOI MAINTENANT, POUR QUI, OBJECTIF, CONTRAINTES, CRITÈRE DE DRIFT. Auteur humain obligatoire, archivé dans .aiad/intents/. Voir l'article Intent Statement.

J

JNSP — « Je Ne Sais Pas »
Verdict d'incertitude (équivalent EN : UNKNOWN) : signal valide, pas un échec. L'agent l'émet quand une zone n'est pas décidable — intention sans humain identifiable, critère non testable, gouvernance ambiguë, annotation @spec absente. Dans le code : TODO-JNSP:, bloqué par le hook pre-commit tant qu'il n'est pas résolu. Exit code CLI : 2. Voir l'article JNSP.

L

Lessons Learned
Section de l'AGENT-GUIDE documentant les erreurs récurrentes de l'agent (au moins 2 occurrences sur des tâches différentes), collectées en rétrospective. À distinguer des Human Learnings.

P

Phase Research
Étape 2 du cycle (/sdd research), intercalée entre l'Intent et la SPEC : elle valide la viabilité de l'intention ancrée dans le code (Discovery obligatoire), pas la qualité de la SPEC. Verdict gradué GO / CONDITIONAL GO / DEFER / NO-GO. Proportionnée : court-circuitable explicitement (et tracé) pour une intention triviale. Voir Phase Research.
PRD — Product Requirement Document
Source de vérité produit : vision, utilisateurs, outcomes. Injecté uniquement en phase de cadrage, jamais en développement. Rédigé via l'assistant /sdd prd.
Product Engineer (PE)
Gardien de l'intention tout au long du cycle : il orchestre des agents IA pour réaliser l'intention sans la trahir, et porte le Context Engineering Budget de chaque session. Voir Les 5 responsabilités.

R

REASONS Canvas
Outil SPDD (Kevlin Henney) optionnel pour structurer la justification d'une SPEC avant le format standard AIAD — proposé par /sdd spec quand l'Intent est complexe ou ambigu. Skill associée : reasons-canvas.
Router
Mécanisme de namespacing des commandes (v1.7) : les 27 commandes sont regroupées sous 3 routers — /sdd <sub>, /aiad <sub>, /aiad-help. Les corps des sous-commandes sont chargés à la demande, ce qui réduit de 94 % le frontmatter chargé à froid. Voir Commandes /sdd.

S

/sdd audit
Commande d'audit qualité du code : conformité SPEC, dette technique introduite, cohérence avec l'AGENT-GUIDE. Rapports persistés dans .aiad/metrics/audit/.
/sdd fact
Commande de capture et qualification d'un écart livré/désiré sans cycle Intent complet — produit un Fact Technique dans .aiad/facts/.
/sdd security
Commande d'audit sécurité : OWASP Top 10, gestion des secrets, permissions des agents (Harness Engineering), conformité AI-ACT / RGPD. Rapports persistés dans .aiad/metrics/security/.
Skill
Bloc de logique réutilisable auto-déclenché (v1.9), stocké dans .claude/skills/ : human-authorship-check, regulatory-veto, drift-detection, sqs-scoring, context-budget, reasons-canvas, ears-validator, traceability. Les commandes complexes composent plusieurs skills sans dupliquer leur logique.
SPEC
Spécification technique détaillée d'une tâche atomique : scope, interface, comportement, cas limites, tests, critère de drift. Une seule SPEC active par session agent. Format prose ou EARS. Voir l'article SPEC.
Spec-Anchored
Positionnement d'AIAD : la SPEC est maintenue comme ancre permanente, synchronisée avec le code à chaque PR. Ni spec-first (spec jetée après la tâche), ni spec-as-source (spec qui génère le code).
Spec Drift
Écart entre l'état d'une SPEC et l'état réel du code. Traité comme un échec de processus, pas comme une erreur d'agent — détecté par la skill drift-detection et le Drift Lock.
SQS — Spec Quality Score
Score de qualité d'une SPEC sur 5 critères (atomicité, précision, testabilité, non-ambiguïté, scope défini) complétés par le Test de l'Étranger. Seuil de Gate : ≥ 4/5 ; un critère non scorable produit ?/5 et une Gate INCONNUE. Voir Métriques — SQS.

T

Test de l'Étranger
Critère 6 du SQS, non scorable : une personne extérieure au projet, sans contexte oral, comprendrait-elle la SPEC et pourrait-elle l'implémenter ? Si la réponse est non, la SPEC repose sur du contexte implicite — elle n'est pas prête pour un agent.
Tier 1
Niveau le plus élevé de la hiérarchie des agents de gouvernance — juste sous la Constitution AIAD. Les 4 agents Tier 1 (AI-ACT, RGPD, RGAA, RGESN) disposent d'un droit de veto sur toute SPEC ou implémentation non conforme, avec politique fail-closed. Voir Gouvernance.
Traçabilité — @intent, @spec, @verified-by, @governance
Matrice machine-vérifiable Intent ↔ SPEC ↔ Code ↔ Tests (v1.10), construite depuis les annotations posées dans le code : @intent INTENT-NNN (0..1), @spec SPEC-NNN-N-slug (1..n, obligatoire sur tout code applicatif), @verified-by (chemin vers un test, 0..n), @governance (référentiels Tier 1, 0..1). Générée par /sdd trace ou npx aiad-sdd trace ; la GitHub Action sdd-trace.yml fait échouer la PR sur gap bloquant (--fail-on-gap). Voir l'article Traçabilité.
Un terme manque ?

Le glossaire suit les évolutions du framework. /aiad-help <mot-clé> retrouve la commande ou le concept associé directement depuis votre agent ; la source de référence reste aiad.ovh.
