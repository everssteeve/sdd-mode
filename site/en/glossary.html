Glossary — the vocabulary of SDD Mode

Every term of the framework, sorted alphabetically — from the founding vocabulary (Intent Statement, SPEC, Drift Lock) to the notions introduced by versions 1.7 through 1.12 (routers, skills, JNSP, EARS, machine-verifiable traceability, Research Phase).

A

AGENT-GUIDE
The agent's permanent context: absolute rules, project conventions, Lessons Learned, Human Learnings and governance cross-references. Injected in condensed form (max 500 tokens) into every session. Since v1.12, it is the upstream source of the multi-runtime rules emitted by emit-rules. See the artifacts.
Anti-Drift Check
End-of-iteration ritual verifying artifact / code synchronisation across the whole project — the periodic application of the Drift Lock. Command: /sdd drift-check.
ARCHITECTURE
The technical-standards document: stack, patterns, structural decisions. Injected condensed (~500 tokens) into every agent session — never in full. Written with the /sdd arch assistant.

C

Context Engineering Budget
The context-absorption capacity of an agent session, owned by the Product Engineer. Rules: one session = one objective, a single active SPEC, 60-70% of the window as the effective maximum, Intent and SPEC at the top of the context. Audited by /sdd context. See the Context Engineering Budget article.
Context Rot
Quality degradation of an LLM before the theoretical limit of its context window: less precise answers, forgotten SPEC constraints, repetitive behaviour. Justifies the 60-70% operational threshold and the SPEC as a stable anchor point.

D

Discovery
Mandatory exploration of the real code during the Research Phase: the viability of an intent is verified in the repository, not in an abstract discussion. The discovery-gate.js hook checks this prerequisite before /sdd spec and /sdd exec.
DoOD — Definition of Output Done
The completion criteria of a task, defined in the SPEC: code, tests, synchronised artifacts. Updating the SPEC is part of it — merged code with an out-of-sync SPEC is not done.
DoOuD — Definition of Outcome Done
The value-oriented counterpart of the DoOD: a feature is only truly "done" when the outcome targeted by the Intent is observed among users — not merely when the code ships.
DORA Metrics
The 4 delivery-performance indicators from the DORA programme: Deployment Frequency, Lead Time for Changes, Change Failure Rate, MTTR. Computed organically by /aiad dora with an Elite / High / Medium / Low rating. See Metrics — DORA.
Drift Lock
PR policy: code and SPEC synchronised in the same commit. Step 7 of the cycle, enforced by the pre-commit hook and by the traceability matrix in CI. See the Drift Lock article.

E

EARS — Easy Approach to Requirements Syntax
Optional SPEC-writing variant (v1.11): SHALL requirements with WHEN / WHILE / IF / WHERE triggers. Enabled by /sdd spec --ears, strictly linted (rules R1-R7) at the Gate: 0 violations → guaranteed +1 on the SQS Testability criterion; ≥ 1 violation → criterion forced to 0. Reserve it for SPECs where ambiguity is expensive — security, payments, compliance, API contracts.
Execution Gate
Checkpoint between the SPEC and the agent launch: no code before the Gate opens, that is SQS ≥ 4/5. Command: /sdd gate. See the Execution Gate & SQS article.

F

Fail-closed
The governance uncertainty policy: when a Tier 1 agent cannot decide, the UNKNOWN verdict counts as a VETO. Block by default rather than let things through — never any silent self-clearance. See Governance.
Flow Metrics
The 5 value-stream health indicators: Cycle Time, Lead Time, Throughput, WIP, Flow Efficiency. Computed by /aiad flow, with a Lead Time breakdown to locate the bottlenecks. See Metrics — Flow.

G

GO / CONDITIONAL GO / DEFER / NO-GO
The graduated verdict of the Research Phase on the viability of an intent: viable (GO), viable under explicit conditions (CONDITIONAL GO), to be postponed (DEFER), not viable (NO-GO). The verdict is decided by the human — the agent prepares the case, it does not rule. See Research Phase.
Governance AGENT-GUIDEs
The 4 Tier 1 regulatory guides holding veto power: AIAD-AI-ACT, AIAD-RGPD, AIAD-RGAA, AIAD-RGESN. Details on the Governance page.

H

Harness Engineering
The Agents Engineer's core practice: configuring, supervising and governing the agents under the minimal necessary permissions principle. Audited by /sdd security.
Human Authorship
Value 7 of the framework: authorship of intent cannot be delegated. Every Intent Statement is written by an identifiable human — if an agent wrote it, the session is invalidated. Checked by the human-authorship-check skill. See the Human Authorship article.
Human Learnings
The AGENT-GUIDE section documenting the gaps between human intent and delivery — fed by the Intent Workshop. Not to be confused with Lessons Learned, which document the agent's errors.

I

Intent Statement
First-order artifact capturing the human WHY in 5 fields: WHY NOW, FOR WHOM, OBJECTIVE, CONSTRAINTS, DRIFT CRITERION. Human author mandatory, archived in .aiad/intents/. See the Intent Statement article.
Intent Workshop
Monthly ritual (60 min max) asking the question: "are we still building what we wanted?". A purely human space — AI agents play no role in it, not even an automatic summary. Prepared by /aiad intention, never facilitated by it. See Loops & rituals.

J

JNSP — "Je Ne Sais Pas — I don't know"
Uncertainty verdict (EN equivalent: UNKNOWN): a valid signal, not a failure. The agent issues it when an area is undecidable — intent with no identifiable human, untestable criterion, ambiguous governance, missing @spec annotation. In code: TODO-JNSP:, blocked by the pre-commit hook until resolved. CLI exit code: 2. See the JNSP article.

L

Lessons Learned
The AGENT-GUIDE section documenting the agent's recurring errors (at least 2 occurrences on different tasks), collected at the retrospective. Not to be confused with Human Learnings.

P

PRD — Product Requirement Document
The product source of truth: vision, users, outcomes. Injected only during the framing phase, never during development. Written with the /sdd prd assistant.
Product Engineer (PE)
Guardian of intent throughout the cycle: orchestrates AI agents to realise the intent without betraying it, and owns the Context Engineering Budget of every session. See The 5 responsibilities.

R

REASONS Canvas
Optional SPDD tool (Kevlin Henney) for structuring the justification of a SPEC before the standard AIAD format — offered by /sdd spec when the Intent is complex or ambiguous. Associated skill: reasons-canvas.
Research Phase
Step 2 of the cycle (/sdd research), inserted between the Intent and the SPEC: it validates the viability of the intent grounded in the code (mandatory Discovery), not the quality of the SPEC. Graduated verdict GO / CONDITIONAL GO / DEFER / NO-GO. Proportionate: it can be explicitly short-circuited (and traced) for a trivial intent. See Research Phase.
Router
Command-namespacing mechanism (v1.7): the 27 commands are grouped under 3 routers — /sdd <sub>, /aiad <sub>, /aiad-help. Sub-command bodies are loaded on demand, cutting the cold-loaded frontmatter by 94%. See /sdd commands.

S

/sdd audit
Code-quality audit command: SPEC conformance, technical debt introduced, consistency with the AGENT-GUIDE. Reports persisted in .aiad/metrics/audit/.
/sdd fact
Command for capturing and qualifying a delivered/desired gap without a full Intent cycle — produces a Technical Fact in .aiad/facts/.
/sdd security
Security audit command: OWASP Top 10, secrets management, agent permissions (Harness Engineering), AI-ACT / RGPD compliance. Reports persisted in .aiad/metrics/security/.
Skill
Reusable, auto-triggered block of logic (v1.9), stored in .claude/skills/: human-authorship-check, regulatory-veto, drift-detection, sqs-scoring, context-budget, reasons-canvas, ears-validator, traceability. Complex commands compose several skills without duplicating their logic.
SPEC
Detailed technical specification of an atomic task: scope, interface, behaviour, edge cases, tests, drift criterion. A single active SPEC per agent session. Prose or EARS format. See the SPEC article.
Spec-Anchored
AIAD's positioning: the SPEC is maintained as a permanent anchor, synchronised with the code on every PR. Neither spec-first (spec discarded after the task), nor spec-as-source (spec that generates the code).
Spec Drift
Gap between the state of a SPEC and the actual state of the code. Treated as a process failure, not an agent error — detected by the drift-detection skill and the Drift Lock.
SQS — Spec Quality Score
Quality score of a SPEC on 5 criteria (atomicity, precision, testability, non-ambiguity, defined scope) completed by the Stranger Test. Gate threshold: ≥ 4/5; an unscorable criterion produces ?/5 and an UNKNOWN Gate. See Metrics — SQS.
Stranger Test
Criterion 6 of the SQS, non-scorable: would a person from outside the project, with no verbal context, understand the SPEC and be able to implement it? If the answer is no, the SPEC rests on implicit context — it is not ready for an agent.

T

Technical Fact
Artifact of /sdd fact: the capture of an observed gap between delivered and desired behaviour, with an impact qualification (functional / security / performance / compliance) and a corrective-action decision traced in .aiad/facts/FACT-NNN.md.
Tier 1
The highest level in the governance-agent hierarchy — just below the AIAD Constitution. The 4 Tier 1 agents (AI-ACT, RGPD, RGAA, RGESN) hold a veto right over any non-compliant SPEC or implementation, with a fail-closed policy. See Governance.
Traceability — @intent, @spec, @verified-by, @governance
Machine-verifiable Intent ↔ SPEC ↔ Code ↔ Tests matrix (v1.10), built from the annotations placed in the code: @intent INTENT-NNN (0..1), @spec SPEC-NNN-N-slug (1..n, mandatory on all application code), @verified-by (path to a test, 0..n), @governance (Tier 1 frameworks, 0..1). Generated by /sdd trace or npx aiad-sdd trace; the sdd-trace.yml GitHub Action fails the PR on a blocking gap (--fail-on-gap). See the Traceability article.
Missing a term?

The glossary tracks the framework's evolutions. /aiad-help <keyword> finds the related command or concept directly from your agent; the reference source remains aiad.ovh.
