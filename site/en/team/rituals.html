Continuous flow

Iterative loops and rituals — rhythm without artificial cadence

AIAD loops are not prescribed fixed-duration cycles: their duration is dictated by the actual complexity of the work. A bug fix: two hours. A new feature: three days. A domain exploration: one week. Each task finds its natural cadence.

Fixed-duration cycles create artificial friction with the velocity of AI-assisted development: they produce either over-commitment or padding. Four characteristics define an AIAD loop: continuous flow (as soon as one feature is integrated, the next one starts), variable duration (complexity dictates duration, not the calendar), dynamic priority (feedback can redirect the next feature) and absolute focus (a single feature at a time per PE). What matters is not following a calendar — it is delivering value continuously.

The initialization phase

Before the first loop, an initialization phase lays the foundations — once per product, or at every major pivot. Without it, agents generate code without context and debt accumulates from day one. It produces seven deliverables: the product vision and Product Goal (PM), the initial PRD (PM), the ARCHITECTURE document (Tech Lead), the AGENT-GUIDE (AE + PE), the Definitions of Done (team), the repository with CI/CD (PE) and the first ready SPEC (PM + PE). The team must be able to start implementation between the fifth and the seventh day. Symmetrical anti-patterns: the initialization that drags on for weeks — no artifact is perfect at the start — or the rushed start without artifacts, whose first weeks cost months of debt.

Loop 1 — Plan

The Plan loop turns a business intent into a specification that agents can implement. Without explicit planning, the PE interprets the PM's intent, the interpretation diverges, and the delivered code does not match the need. Five steps: the PM presents the next priority, the PE questions and clarifies, the team decides on the decomposition, writes the SPEC collaboratively, then validates it. It produces a SPEC ready for implementation, defined Outcome Criteria and a shared PM/PE understanding. Indicators: the agent can implement without clarification, expected tests are 100% explicit, all edge cases are documented. Anti-patterns: the four-hour planning marathon as much as the three-line "Add login" SPEC — two anti-investments.

Loop 2 — Implement

This is where value materialises: a PE who orchestrates well produces more than a traditional team of five developers. The loop unfolds in seven steps: prepare the context (SPEC + ARCHITECTURE + AGENT-GUIDE, within the Context Engineering Budget), ask the agent for a plan before coding, validate continuously (compilation, linting, types), generate the tests, iterate if needed — three times maximum —, finalise (green tests, local commit), then carry out the artifact Sync micro-phase.

Artifact Sync micro-phase — 15 minutes, non-negotiable

Under the PE's responsibility, four questions: does the SPEC reflect the implemented behaviour? Is the Drift Criterion still valid? Was a notable agent error fixed manually (→ Lessons Learned)? Has a new convention emerged (→ AGENT-GUIDE patterns)?

Indicators: first-time-correct code > 70%, generated/manual code ratio > 80/20, coverage > 80% backend and 70% frontend. Anti-pattern: the PE who fixes 80% of the generated code — a signal of a deficient SPEC or AGENT-GUIDE, not of a bad agent.

Loop 3 — Validate

Code that compiles is not necessarily code that works; code that works is not necessarily code that delivers value. Validation unfolds in five steps with distinct owners:

The five steps of the Validate loop
StepOwnerContent
Technical validationPECI, coverage, linting, DoOD
Functional validationQATests, acceptance criteria
Usability validationQA + PMUX, accessibility, performance
Business validationPMIntent respected, outcomes
DecisionTeamVALIDATED, FIXES or REJECTED

It produces a validation report, a list of minor fixes where applicable, and the green light for integration. Indicators: first-try validation > 70%, zero critical bugs, fewer than three minor bugs per feature. Anti-patterns: the rushed "it seems to work" validation (hidden debt) and the endless QA/PE ping-pong (needless friction).

Loop 4 — Integrate

Undeployed code has no value. The loop unfolds in eight steps: code review (self or peer depending on criticality), pull main and resolve conflicts, push, PR and CI/CD, staging then production deployment, post-deployment verification, CHANGELOG and AGENT-GUIDE update, Anti-Drift Check, and context cleanup with ticket closure.

Anti-Drift Check — 10 minutes, three questions

Before closing the ticket, the PE answers for each SPEC used: does the delivered behaviour match exactly what the SPEC describes? Is the list of impacted files still exhaustive? Does the DRIFT CRITERION of the parent Intent Statement signal a gap? If any answer is no, the SPEC is updated — committed in the same PR as the code: that is the Drift Lock.

Four deployment strategies are available: Continuous Deployment (non-critical features), Staged Rollout (major features), Feature Flags (experiments, A/B tests) and Manual Release (critical, compliance). AIAD recommendation: Continuous Deployment with Feature Flags. Indicators: merge → production in under one hour (ideally fifteen minutes), rollback < 5%, zero downtime.

Intentional synchronisations

Synchronisations are not ceremonies: they are moments of collective decision triggered by a real need, timeboxed, actionable and documented. Each one is supported by an /aiad command that prepares the data — never replaces the conversation.

  • Daily standup (optional, 15 min max — /aiad standup): structured around three SDD questions — which SPEC today? what Context Engineering Budget? any drift to report? Over 80% of blockers resolved within the day.
  • Demo & Feedback (bi-weekly, 30-60 min — /aiad demo): the demonstration is structured by the Intent Statements — first the original why, then what was delivered — before qualitative user feedback and backlog adaptation. Target: more than three actionable insights per session.
  • Tech Review (bi-weekly to monthly, 1-2 h — /aiad tech-review): checks the coherence between ARCHITECTURE.md and the actual code — pattern drift, undocumented dependencies, ADRs to write, technical debt, agent ecosystem review.
  • Retrospective (weekly or bi-weekly, 45-60 min — /aiad retro): a classic retrospective, then an AI retrospective — collecting Lessons Learned (recurring agent errors) and Human Learnings (intent/delivery gaps) fed into the AGENT-GUIDE, along with the iteration metrics (average SQS, drifts detected, first-time success rate). One to three actions maximum, with owners.

The Intent Workshop — a purely human space

The Intent Workshop (Atelier d'Intention) is AIAD's fundamental synchronisation: monthly, 60 minutes, the whole team. The other synchronisations ask "what did we deliver?", "does it work?"; this one alone asks the question that the agent cannot ask in our place: "Are we still building what we really wanted to build?" Four phases: reading the Intent Statements out loud (10 min), confronting them with reality (20 min), gap hypotheses (20 min), alignment or revision decision (10 min).

Non-negotiable rule: no AI

Agents have no role in this workshop — no automatic summary, no AI analysis of the gaps. The /aiad intention command serves only to prepare the briefing pack (the month's Intent Statements, alignment metrics, Human Learnings) — never to intervene during. It is a purely human space, including in --fast mode.

Target: intent/delivery alignment rate > 80%, two to five documented gap hypotheses, 100% participation. Anti-pattern: the workshop drifting into a technical review or a bug post-mortem — the questions stay at the level of intent.

Strategic Alignment synchronisation

Monthly, 1 h 30 to 2 h (/aiad sync-strat), with the PM, PE, Tech Lead, QA, stakeholders and Supporters: outcome review, Product Goal review, backlog prioritisation — including what we decide not to do —, Supporter feedback on organisational obstacles, documented decisions. It relies on the DORA and Flow metrics to adjust priorities.

Loop and synchronisation anti-patterns

  • "We plan everything at the start of the week." Back to batch mode: priorities change but the plan is frozen. Plan just before implementing, while the context is fresh.
  • "Implementation always takes longer than expected." Insufficient SPECs or a poorly prepared agent context: thirty extra minutes of planning save hours of implementation.
  • "We validate at the end, just before the release." Bugs pile up, validation becomes a bottleneck. Validate each feature immediately.
  • "We hold every sync every week." Meeting overload: adapt the frequency to the real need, space them out as the team matures.
  • "Nobody prepares the syncs." Inefficient improvisation sessions: every synchronisation has an owner who prepares the agenda and the inputs.
