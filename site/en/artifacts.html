Documentation hierarchy

The artifacts — the team's collective memory

AIAD artifacts are not bureaucracy: they are tools for thinking and communicating. They make visible what would otherwise stay implicit, align the team on a shared understanding, and give AI agents the context they need to perform. Without them, the context is lost at every session.

Thinking tools, not paperwork

A good artifact is recognised by four qualities: it is actionable (can we act from this document?), living (does it evolve with understanding?), minimal (does it contain just enough, no more?) and collaborative (was it co-created, not dictated?). What matters is not having perfect documents — it is having useful documents. Without an AGENT-GUIDE, agents produce generic code; without ARCHITECTURE, inconsistent code; without a SPEC, incorrect code.

The documentation hierarchy

The artifacts are organised in a strict hierarchy. The rule is absolute: never contradict a higher level.

AIAD Constitution (immutable values)
  └── Tier 1 Governance Agents (veto power)
       └── PRD (product vision)
            └── ARCHITECTURE (technical standards)
                 └── AGENT-GUIDE (permanent context)
                      └── SPEC (per-task activation)

This hierarchy is also the management tool of the Context Engineering Budget: strategic context at the top (injected during framing), permanent context in the middle (condensed), task context at the bottom (one SPEC at a time). In a conflict between a SPEC and a governance agent, governance prevails — always.

Intent Statement — the archived WHY

The Intent Statement is the first-order artifact: it captures the human intent behind each feature in five fields (WHY NOW, FOR WHOM, OBJECTIVE, CONSTRAINTS, DRIFT CRITERION), with a mandatory human author — the Human Authorship principle. It precedes and transcends the SPEC, and it is archived in .aiad/intents/. Six months from now, it is the one that will answer the question "what did we want to accomplish, and why?".

PRD — the expected value

The Product Requirement Document clarifies why and what to build before the how. It contains the context and the problem, the measurable Outcome Criteria, the personas and use cases, the deliberate out-of-scope, the trade-offs and the risks. Five practices are non-negotiable: start from the problem (not the solution), define the success criteria before building, write it with the team, keep it living, and illustrate it (wireframes, flows).

Anti-pattern

The fifty-page river of a PRD that nobody reads, or the vague PRD along the lines of "Improve the user experience". Both are useless. And on the agent side: the full PRD is only injected during the framing phase, never during development.

ARCHITECTURE — the coherence reference

The ARCHITECTURE document defines the technical standards that agents and PEs must respect, in nine sections: architectural principles, overview, justified tech stack, project structure, code conventions, patterns and best practices, security, performance, and ADRs (Architecture Decision Records). It is evolutive, justified, pragmatic (YAGNI), visual and actionable. In an agent session, it is never injected in full: a permanent condensed version of 500 tokens maximum is enough — the rest is consulted on demand.

AGENT-GUIDE — the agent's permanent context

The AGENT-GUIDE is the most operational document of the framework: injected into every session, it turns a generic agent into a project agent. It contains the project identity, the absolute ALWAYS / NEVER rules (e.g.: always synchronise SPEC + code in the same PR; never push secrets), the code conventions, the domain vocabulary, the patterns and anti-patterns — and two complementary learning sections:

Lessons Learned vs Human Learnings
SectionQuestion askedUpdated
Lessons Learned"What did the agent get wrong?" — recurring agent errors and corrections appliedEnd of each iteration (5 min, not optional)
Human Learnings"What did the human express poorly?" — failures in expressing the intentReviewed at the monthly Intent Workshop

A guide with a hundred rules is less effective than a guide with ten well-chosen ones: the AGENT-GUIDE is bounded by the Context Budget (condensed ≤ 500 tokens per session).

SPECs — the agent's work instruction

A SPEC bridges the intent (PRD, Intent) and the agent's implementation. A quality SPEC makes it possible to generate 80%+ of correct code on the first pass — it is the framework's best-ROI investment. Four criteria define it: atomicity, precision, testability, completeness — scored by the SQS at the Execution Gate (≥ 4/5 required). Each SPEC carries its own Drift Criterion and stays synchronised with the code on every PR.

The Definitions of Done

The Definition of Output Done (DoOD) sets the standard for an increment to be shippable, in six categories: technical, security, performance, functional, deployment, review. The principle is non-negotiable: all criteria or nothing — "90% Done is not Done".

The Definition of Outcome Done (DoOuD) measures whether the expected value was realised: User Outcomes (NPS, adoption, retention), Business Outcomes (MRR, conversions, efficiency) and Learning Outcomes (hypotheses validated or invalidated, insights). The output is the means; the outcome is the goal — a feature shipped but not adopted is not a success.

├── PRD.md                  ← Product vision (injection: framing only)
├── ARCHITECTURE.md         ← Technical standards (permanent condensed version)

├── gouvernance/            ← Tier 1 agents with veto power

│   ├── AIAD-RGPD.md

│   └── _index.md

│   └── _index.md

├── metrics/

Artifact anti-patterns

Three ways to kill your artifacts

The hundred-page guide — Context Budget exceeded, the agent ignores most of the rules. The frozen SPEC — "validated, we don't touch it anymore": five iterations later, that's spec drift and the agent works on a wrong basis. Injecting everything at once — the agent receives everything, understands little: that's context rot. AGENT-GUIDE for the permanent, SPEC for the task — never mix the two.
