Cycle step 4 · /sdd gate

Execution Gate — SQS ≥ 4/5 or nothing

The Execution Gate is the checkpoint between a written SPEC and launching the agent. The rule is simple and admits no exception: no code before the Gate is open. The Spec Quality Score (SQS) must reach 4/5, and the Stranger Test must be documented.

Why a gate before the code

Handing a mediocre SPEC to an agent does not save time — it wastes it. Fixing a SPEC takes ten minutes; fixing code generated from a bad SPEC takes hours. The Gate embodies that economic trade-off: it forces the investment where the ROI is highest, before the first line is generated.

The Gate is run by the sqs-scoring skill, fed by ears-validator. Three possible outcomes: OPEN, CLOSED (with a remediation plan) — or UNKNOWN when the score is undecidable.

The five Spec Quality Score criteria

The 5 SQS criteria
#CriterionQuestion asked
1AtomicityA single functional responsibility? (1 SPEC = 1 PR)
2Precise interfacesExplicit types and signatures — no vague "return an object"?
3TestabilityTest cases included and linked to the PRD's Outcome Criteria?
4Non-ambiguityZero vague adverbs — "correctly", "appropriately", "quickly"?
5Defined scopeExhaustive impacted files and a filled-in Drift Criterion?

Each criterion is worth 0 or 1 point. Score ≥ 4/5: the Gate opens and the SPEC moves to ready status. Score < 4/5: back to revision with a remediation plan that is a deliverable in its own right — proposed replacement text and estimated effort, criterion by criterion. Two special routes: Atomicity at 0 → /sdd split; three consecutive failures → go back to /sdd intent, the problem is upstream.

Criterion 6 — the Stranger Test

A sixth, non-scorable criterion completes the checklist: "If I showed this SPEC to someone who doesn't know the project, would they understand why we are doing it?" It does not block the Gate if the answer is negative, but the question must be asked, and the answer — two sentences at most, written by a human — added to the SPEC before continuing:

## Context for an outside reader
[Two sentences. Written by a human. Before the Execution Gate.]

It is the Feynman principle applied to specifications: you only truly understand something when you can explain it simply. If the two sentences are impossible to write, the human context is insufficient — go back to the Intent Statement before continuing.

Strict EARS mode

When the SPEC declares Format : EARS in its header (see the EARS variant), the ears-validator linter switches to strict mode and mechanically affects criterion 2:

Effect of the EARS linter on the SQS
R1–R7 lint resultEffect on SQS criterion 2 (Testability)
0 violationsGuaranteed +1 bonus (capped at 1/1)
≥ 1 violationCriterion forced to 0 — Gate closed, with a criterion-by-criterion R1–R7 rewording table

In prose format, the linter remains advisory: it flags ambiguities without altering the score.

UNKNOWN Gate — the SQS fail-closed

When one or more criteria are not scorable — testability impossible to judge without a known test stack, consistency unverifiable because a dependent SPEC is unreadable, atomicity undecidable because the parent Intent is unstated — the Gate exits neither OPEN nor CLOSED: it exits UNKNOWN, with a ?/5 score. This is a JNSP state (Je Ne Sais Pas — "I don't know") and it is fail-closed: no agent execution may start.

Gate: UNKNOWN (JNSP verdict)
SQS: ?/5
Non-scorable criteria:
  - Criterion 2 (Testability): reason = <what prevents scoring>
  - Criterion 4 (Consistency): reason = <what prevents scoring>
Question for the human: <actionable question>
Forbidden

Inventing a default score "so as not to block" is exactly the behaviour the UNKNOWN Gate prevents. The SPEC stays in draft or review until the human decides.

Preparing the Context Engineering Budget

If the Gate opens, it does more than rubber-stamp the SPEC: it prepares the Context Engineering Budget of the upcoming agent session — the list of files to inject, the estimated token total, and a check against the 50K-token threshold beyond which context rot looms. The budget is then re-verified at launch by /sdd exec.

Empirical validation

The Execution Gate is not a comfort ritual: it answers a measured bias. SOTA agents exhibit an action bias of 35 to 65% — the tendency to produce code rather than flag that a requirement is ambiguous or infeasible. The Gate imposes the human stopping point the agent does not impose on itself. The evidence synthesis is consolidated in docs/legitimation/execution-gate-evidence.md.

Guided mode: "grill me"

In --guided, the grill-me skill turns the Gate into an interactive interrogation: one question at a time, the agent proposes its recommended answer, the human validates or corrects. Never a static form — the decision to open the Gate remains a human act.
