# Note de release — aiad-sdd **v1.18.0**

> **Date** : 2026-06-09 · **Type** : minor (SemVer) · **Rupture** : aucune — tout est **additif et opt-in**.
> **Cible** : Claude Code v2.1.168 · Opus 4.8 · effort `xhigh`/`max`.
> Cette note est volontairement **pédagogique** : elle explique le *pourquoi* avant le *quoi*, puis donne le *comment*.

---

## 1. En une phrase

La v1.18.0 fait passer SDD Mode de **« le modèle *devrait* respecter les règles »** à **« le modèle *ne peut pas* les contourner »** : chaque règle critique (gouvernance, JNSP, Drift Lock, qualité de spec) devient une **primitive vérifiée par le harness**, et chaque verdict est calculé par un **script déterministe** plutôt que par le jugement libre de l'IA.

C'est l'aboutissement d'une roadmap complète (P0 → P3 + garde-fous §4) dérivée de l'analyse `docs/analyse-claude-code-best-practice.md`.

---

## 2. Le concept central : *advisory → enforced*

### Le problème

Jusqu'ici, SDD Mode reposait surtout sur du **texte** : le `CLAUDE.md` disait à l'agent « ne code pas sans SPEC », « respecte les vetos de gouvernance », « dis JNSP si tu ne sais pas ». Mais un texte est une **suggestion**. Un modèle peut l'ignorer, l'oublier en fin de contexte (*lost in the middle*), ou le contourner par excès de confiance. Le coût caché réapparaît plus tard : drift, bug de prod, violation réglementaire.

### La bascule

On migre chaque règle critique vers une **primitive du harness** que le modèle subit, pas qu'il interprète :

| Mécanisme harness | Rôle | Peut-on l'ignorer ? |
|-------------------|------|---------------------|
| **Hook `PreToolUse`** | bloque un `git commit` (JNSP non résolu, veto Tier 1) | Non — le commit est refusé |
| **Hook `Stop`** | vérifie le Drift Lock à la fin du tour | Non — relance tant qu'il y a un gap |
| **Hook `UserPromptSubmit`** | rappelle le prérequis Discovery avant `/sdd spec` | Non — injecté dans le contexte |
| **Subagents Tier 1 read-only** | gouvernance avec droit de veto, sans accès `Edit`/`Write` | Non — outils retirés |

### Le corollaire : *computation off-context*

Un verdict (PASS / FAIL / JNSP) ne doit **jamais** venir du jugement libre du modèle — il n'y a aucun déterminisme garanti (pas de *seed*, variance de serving, routing MoE). Tous les verdicts sont donc produits par des **scripts CLI déterministes** (`lib/*.js`), avec :

- des **exit codes stables** : `0` = PASS · `1` = FAIL · `2` = JNSP *(décision humaine requise — ce n'est pas une erreur)* ;
- une **sortie JSON validée par schéma** (`.aiad/schema/verdicts/*.schema.json`).

> **À retenir** : ce que le modèle *propose*, un script le *vérifie*. La même enveloppe de verdict est lue par l'humain, injectée au modèle et consommée par le hook — une seule source de vérité.

### Le contre-pouvoir : `JNSP` (Je Ne Sais Pas)

Dire « je ne sais pas faire » est un **signal valide**, pas un échec. Plutôt qu'une réponse confiante mais inventée, l'outil émet `JNSP` (exit `2`) quand l'intention n'est pas humaine, qu'un critère n'est pas testable, ou qu'une gouvernance n'est pas décidable (fail-closed). Inventer est un coût caché ; `JNSP` le rend visible **avant** qu'il ne devienne un bug.

## 3. Le cycle SDD, désormais outillé de bout en bout

```
Intent → Research (GO/NO-GO) → SPEC → Execution Gate (SQS ≥ 4/5) → Exécution → Validation → Drift Lock
```

Chaque flèche a maintenant un **verdict déterministe** et, si on le souhaite, un **nœud de graphe** qui empêche de sauter l'étape.

---

## 4. Nouveautés par phase

Pour chaque bloc : **le problème → ce qui change → comment l'utiliser**.

### P0 — Rendre la gouvernance *enforced* (§3.1 → §3.4)

**Le problème** : vetos, JNSP et Drift Lock n'étaient que du texte.

**Ce qui change** :
- **§3.4 Verdicts machine-vérifiables** — `lib/verdict.js` : contrat unique (`PASS|CONDITIONAL|FAIL|JNSP`), exit `0/1/2`, validateur JSON Schema zero-dep. C'est la **fondation** de tout le reste.
- **§3.2 JNSP enforced** — hook `PreToolUse` (`jnsp-scan.js`) : un `git commit` contenant un `TODO-JNSP:` non résolu est **refusé**.
- **§3.3 Drift Lock enforced** — hook `Stop` (`drift-lock.js`) : à la fin du tour, `trace` recalcule la traçabilité ; un gap bloquant relance l'agent.
- **§3.1 Vetos Tier 1 enforced** — subagents de gouvernance **read-only** générés par `emit-rules` (AI-ACT / RGPD / RGAA / RGESN), hook `veto.js` *fail-closed* (`UNKNOWN` = VETO), et `managed-settings.json` org.

**Comment** : tout s'installe via `npx aiad-sdd init` (hooks + settings + subagents). Vérification : `npx aiad-sdd veto --diff <patch> --output-format verdict`.

### P1 — Combler les 2 trous du cycle + alléger le contexte (§3.5 → §3.7)

- **§3.5 Phase Research + Discovery** — une étape `Research (GO/NO-GO)` s'intercale entre l'Intent et la SPEC. Elle valide la **viabilité de l'intention**, ancrée dans le code réel (Discovery obligatoire — anti « specs-to-code naïf »). Verdict gradué `GO | CONDITIONAL GO | DEFER | NO-GO`, **tranché par l'humain**.
  - `npx aiad-sdd research <RESEARCH-id>` · `npx aiad-sdd discovery-check <INTENT-id>`
- **§3.6 Exécution phasée + mini-gates** — une SPEC se découpe en tranches ; chaque tranche a son **mini-gate** (`PASS | CONDITIONAL | FAIL | JNSP`). Le 3ᵉ verdict `CONDITIONAL` permet d'avancer en traçant une dette à lever.
  - `npx aiad-sdd exec-status <SPEC-id>` · `npx aiad-sdd mini-gate <SPEC-id> --phase N`
- **§3.7 Budget d'instructions : push → pull** — la gouvernance advisory n'est plus *poussée* en permanence : elle est *tirée à la demande* via `.claude/rules/*.md` à frontmatter `paths:` (RGPD ne se charge que sur les zones de données, RGAA que sur l'UI…). Réglages de budget natifs dans `settings.json`.

### P2 — Capitalisation, robustesse, anti-bruit (§3.8 → §3.10)

- **§3.8 Memory native** — on ne mémorise une leçon que si elle **récurre** (*from logs, pas from one transcript* : « make the button pink » ≠ « tous les boutons roses »), et **jamais sans auteur humain**. Auto-curation > 200 lignes. Cycle anti *dock rot* pour sortir les artefacts livrés du contexte chaud.
  - `npx aiad-sdd memory propose` → `memory promote --auteur "Nom"` · `npx aiad-sdd archive --delivered`
- **§3.9 Cycle SDD comme graphe de Tasks** — le cycle devient un **graphe `blockedBy`** : une étape dont la précédente n'est pas terminée **ne peut pas démarrer**. Persistant et *crash-recoverable* (lu par `/sdd resume`).
  - `npx aiad-sdd cycle init <INTENT-id>` · `cycle step <INTENT-id> GATE PASS` · `cycle next <INTENT-id>`
- **§3.10 Canary suite** — « *frozen weights ≠ frozen behavior* » : un set de cas figés rejoué quotidiennement distingue une **régression réelle** d'un **bruit de serving** (±8-14 %). Et alignement documentaire **Opus 4.7 → Opus 4.8**.
  - `npx aiad-sdd canary --output-format verdict`

### P3 — Confort, observabilité, distribution (§3.11 → §3.13)

- **§3.11 Observabilité native** — une **statusLine** live (`SPEC active │ Gate ✅ │ étape │ ctx % │ effort │ coût`), la **mesure d'usage des skills** (hook `PreToolUse(Skill)`), le **`Co-Authored-By`** garanti par les settings, et un **scaffolding OpenTelemetry** opt-in.
  - `echo '{}' | npx aiad-sdd statusline`
- **§3.12 Cross-model review « additive-only »** — un **modèle tiers** (Codex/Gemini) review le diff en *contexte frais* et insère des **Findings cités sans jamais réécrire**. Le verdict reste déterministe ; des findings hauts non résolus forcent **au plus** `CONDITIONAL` (jamais un FAIL inventé).
  - `npx aiad-sdd cross-model prompt <SPEC-id> --reviewer codex` → `cross-model merge <SPEC-id>`
- **§3.13 Distribution & pilotage** — SDD devient un **plugin Claude Code** (`.claude-plugin/`), des recettes **`/goal`** bouclent sur un verdict CLI déterministe, et des **toggles de hooks** par environnement (`hooks-config.json` + `.local.json`) — **sans jamais** pouvoir désactiver le veto de gouvernance Tier 1.
  - `npx aiad-sdd hooks-config show` · `/plugin install aiad-sdd`

### §4 — Garde-fous de conception (la philosophie qui borne le reste)

Cinq principes pour **éviter la sur-ingénierie** et rester aligné sur l'évolution des modèles :

| | Garde-fou | Outil |
|---|-----------|-------|
| **GF1** | SDD = *agentic engineering* formalisé (Human Authorship + Verifiability) | doctrine `README.md` |
| **GF2** | Garder le code en boucle (Discovery obligatoire) | checklist `CONTRIBUTING.md` |
| **GF3** | Léger par défaut, lourd seulement si l'ambiguïté coûte cher | `aiad-sdd proportionality <INTENT-id>` |
| **GF4** | Gate humain interactif « grill me » (1 question/tour + recommandation) | skill `grill-me` |
| **GF5** | Prévoir des règles qui se suppriment | `aiad-sdd sunset` (+ `doctor`) |

## 7. Compatibilité & migration

- **Aucune rupture.** Tout est **additif** : les commandes, hooks et artefacts existants fonctionnent comme avant.
- **Opt-in** : OTel, cross-model, canary génératif et toggles sont désactivés/neutres par défaut.
- **Garde-fou de sécurité** : un toggle d'environnement **ne peut pas** désactiver un veto de gouvernance Tier 1 (sauf `allowDisableGovernance`, qu'un `managed-settings.json` org peut interdire).
- **Annotations** : le code applicatif doit porter `@spec SPEC-NNN-N-slug` (et au besoin `@intent`, `@verified-by`, `@governance`) — déjà la convention depuis la v1.10, désormais **mesurée** par les hooks.

### Mettre à jour

```bash
# Mise à jour du paquet
npm i -D aiad-sdd@latest          # ou: npx aiad-sdd self-update

# Régénérer hooks / settings / subagents / règles dans un projet existant
npx aiad-sdd init                 # idempotent : ajoute sans écraser
npx aiad-sdd emit-rules           # resynchronise CLAUDE.md / AGENTS.md / .cursor / .codex / GEMINI

# Vérifier l'état
npx aiad-sdd doctor
npx aiad-sdd hooks-config show
```
