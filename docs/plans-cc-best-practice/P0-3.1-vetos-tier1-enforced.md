# Plan — §3.1 Vetos Tier 1 → primitives harness (pas du texte)

> **Recommandation** : §3.1 de `analyse-claude-code-best-practice.md`
> **Priorité** : P0 · Impact 🔴 Critique · Effort Moyen
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

Aujourd'hui les 4 agents de gouvernance Tier 1 (AI-ACT, RGPD, RGAA, RGESN) ont un *droit de veto* qui n'existe **que dans le texte** de `.aiad/gouvernance/*.md` et de `CLAUDE.md`. Le modèle *devrait* respecter le veto, mais rien ne l'y *contraint*. L'analyse (§1, §6.2) impose de transformer ce veto en primitive **enforced** : « le modèle ne peut pas le contourner ». La paternité humaine reste intacte — c'est l'humain qui définit les règles de veto ; le harness ne fait qu'en garantir l'exécution.

## 2. État actuel SDD Mode

| Élément | Fichier réel | Nature |
|---------|--------------|--------|
| Définition des 4 agents | `.aiad/gouvernance/{AIAD-AI-ACT,AIAD-RGPD,AIAD-RGAA,AIAD-RGESN}.md` + `_index.md` | Texte advisory |
| Skill d'application | `templates/.claude/skills/regulatory-veto/SKILL.md` | Inline, advisory |
| Lint CLI gouvernance | `lib/governance-lint.js` (`lintGouvernance`), `lib/governance.js` | Déterministe mais non bloquant au niveau harness |
| Politique fail-closed | `CLAUDE.md` : « UNKNOWN = VETO » | Texte |
| Settings émis | `templates/.claude/settings.json` (uniquement `SessionStart`) | Pas de `deny`, pas de hook de veto |

**Manque** : aucun `deny`/`hard_deny`, aucun subagent `.claude/agents/*.md`, aucun hook `PreToolUse`/`Stop` de type `agent` qui bloque réellement.

## 3. Cible enforced (3 leviers cumulables)

1. **`deny` rules + `autoMode.hard_deny`** dans un `managed-settings.json` (niveau org) : précédence absolue, non-overridable, `hard_deny` non lu depuis les settings projet partagés (anti-injection).
2. **Agents de gouvernance = vrais subagents** `.claude/agents/AIAD-*.md` : `description: PROACTIVELY`, `disallowedTools` (un agent RGPD ne peut **pas écrire**), `memory: project` (jurisprudence versionnée), `paths:` ciblés.
3. **Hook `PreToolUse`/`Stop` de type `agent`** invoquant `regulatory-veto`/`lib/governance-lint.js` sur le diff et **bloquant** (exit 2), sur le modèle « Execution Contract + fail-closed + allowlist » du `weather-agent`.

## 4. Conception détaillée

### 4.1 Couche `deny` managed (levier 1)
Créer un template `templates/managed-settings.json` (+ doc d'installation org dans `docs/`) :
```jsonc
{
  "permissions": {
    "deny": [
      "Edit(.aiad/gouvernance/**)",          // personne ne réécrit le veto sans process
      "Write(.aiad/gouvernance/**)"
    ]
  },
  "autoMode": {
    "hard_deny": [
      "Bash(git commit * --no-verify*)",      // pas de bypass du pre-commit veto
      "Bash(git push * --force*)"
    ]
  }
}
```
Émis **uniquement** sur opt-in org (`aiad-sdd init --managed`), jamais imposé par défaut (Sobriété Intentionnelle).

### 4.2 Subagents de gouvernance (levier 2)
Nouveau dossier `templates/.claude/agents/` avec 4 fichiers générés par `emit-rules` depuis la source `.aiad/gouvernance/` (source unique préservée). Frontmatter type :
```yaml
---
name: AIAD-RGPD
description: PROACTIVELY review any change touching personal data for RGPD compliance. Fail-closed: UNKNOWN = VETO.
tools: Read, Grep, Glob          # JAMAIS Edit/Write/Bash → veto = lecture seule
disallowedTools: Edit, Write, Bash, NotebookEdit
model: inherit
memory: project                  # .claude/agent-memory/AIAD-RGPD/ — jurisprudence versionnée
paths: ["**/*pii*", "**/*user*", "**/auth/**", "**/*personal*"]
---
# Execution Contract (non-negotiable)
- Tu ne peux pas écrire de fichier. Ton verdict est consultatif au modèle mais bloquant au hook.
- Verdict ∈ {CONFORME, NON-CONFORME, UNKNOWN}. UNKNOWN ⇒ VETO (fail-closed).
- Tu cites une evidence (fichier:ligne) pour chaque verdict.
```
Le `Explore` built-in (haiku, read-only) est le modèle de référence : rapide et **incapable d'écrire** (§2.1).

### 4.3 Hook bloquant de type `agent` (levier 3)
Ajouter dans `templates/.claude/settings.json` :
```jsonc
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "if": "Bash(git commit *)",
        "hooks": [{
          "type": "agent",
          "agent": "AIAD-RGPD",          // ou un routeur regulatory-veto
          "statusMessage": "Veto Tier 1 en cours…",
          "timeout": 120
        }]
      }
    ]
  }
}
```
Le handler `agent` lit le diff (`git diff --cached`), applique `lib/governance-lint.js`, et renvoie `permissionDecision: "deny"` + exit 2 si veto. **Le calcul du verdict reste déterministe** (CLI), l'agent ne fait qu'orchestrer.

### 4.4 Source unique préservée
`lib/emit-rules.js` génère `.claude/agents/AIAD-*.md` **et** les fragments CLAUDE.md/AGENTS.md depuis `.aiad/gouvernance/`. Le workflow `aiad-emit-rules-check.yml` est étendu pour vérifier la non-divergence des agents émis.

## 5. Étapes d'implémentation séquencées

1. **Spec** : rédiger SPEC-NNN-veto-enforced (format EARS — domaine réglementaire = ambiguïté coûteuse).
2. Étendre `lib/emit-rules.js` : nouvelle cible `agents` → `templates/.claude/agents/AIAD-*.md` (générée depuis `.aiad/gouvernance/`).
3. Créer le template `templates/managed-settings.json` + section doc `docs/` (installation org, opt-in).
4. Ajouter le bloc `hooks.PreToolUse` (type `agent`) dans `templates/.claude/settings.json`.
5. Exposer une sous-commande CLI `aiad-sdd veto --diff <ref> --json` (réutilise `lintGouvernance`) → exit 0/1/2.
6. Brancher le handler `agent` du hook sur cette commande.
7. Étendre `aiad-emit-rules-check.yml` pour couvrir `.claude/agents/`.
8. Drift Lock : SPEC + code dans la même PR.

## 7. Critères d'acceptation (EARS)

- **WHEN** un `git commit` touche un fichier matché par `paths:` d'un agent Tier 1, **the system SHALL** exécuter le veto correspondant avant la création du commit.
- **IF** le verdict d'un agent Tier 1 est `NON-CONFORME` **OR** `UNKNOWN`, **the system SHALL** bloquer le commit avec exit code 2.
- **WHERE** un agent de gouvernance est défini, **the system SHALL** lui interdire tout outil d'écriture (`Edit`, `Write`, `Bash`).
- **the system SHALL** générer les agents `.claude/agents/AIAD-*.md` depuis la source unique `.aiad/gouvernance/` sans divergence (vérifié en CI).

## 8. Tests & vérification

- Tests unitaires `lib/veto.js` : diff PII → NON-CONFORME ; diff neutre → CONFORME ; diff ambigu → UNKNOWN (exit 2).
- Test d'intégration : `emit-rules` produit 4 agents conformes à la source ; divergence ⇒ CI rouge.
- Test harness (manuel/CI headless) : un commit sur fichier PII sans annotation `@governance` est bloqué.
- Test anti-injection : un `settings.json` projet tentant d'overrider un `hard_deny` est ignoré.

## 9. Risques, dépendances, rollback

- **Risque** : faux positifs bloquant des commits légitimes → prévoir `aiad-sdd veto --explain` + bypass tracé (`AIAD_SKIP_VETO=1`, loggé JSONL comme `hook-runs.jsonl`).
- **Dépendance** : §3.4 (verdicts JSON/exit codes) — à livrer avant ou avec.
- **Compatibilité** : `managed-settings.json` est opt-in org ; les projets solo gardent le comportement advisory + hook local.
- **Rollback** : désactiver le bloc `PreToolUse` via `hooks-config.json` (toggle `disablePreToolUseHook`), sans toucher au code.
