# Plan — §3.10 Canary suite + alignement des références modèles

> **Recommandation** : §3.10 de `analyse-claude-code-best-practice.md`
> **Priorité** : P2 · Impact 🟡 Moyen · Effort Faible
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

« Frozen weights ≠ frozen behavior » (§2.2) : variance mesurée **±8-14 %** (bugs infra, routing MoE, post-training silencieux, **context pollution = cause #1**). Sans point de référence stable, impossible de distinguer une **régression réelle** d'un **bruit serving** — et `/aiad retro` risque de tirer des conclusions du bruit. L'analyse recommande : (a) une **canary suite** — set fixe d'Intents/SPECs rejoué quotidiennement ; (b) **pin des snapshots** + séparation « qualité modèle » / « fiabilité serving » ; (c) **aligner les références** documentaires : Opus 4.7/1M → **Opus 4.8, effort `xhigh`/`max`, Claude Code v2.1.168**.

## 3. Cible

1. **Canary suite** : `.aiad/canary/` = set figé d'Intents+SPECs+verdicts attendus, rejoué quotidiennement (`aiad-sdd canary`), comparé à une baseline.
2. **Séparer signal/bruit** : un écart < seuil de variance (±8-14 %) = bruit ; au-delà = régression à investiguer.
3. **Alignement des réfs** : remplacer Opus 4.7 → 4.8 / effort `xhigh`/`max` / v2.1.168 dans toute la doc.

## 4. Conception détaillée

### 4.1 Canary suite
- `.aiad/canary/cases/*.md` : N cas figés (Intent + SPEC) avec **verdict attendu déterministe** (SQS, gate, trace) — comme ce sont des **verdicts CLI déterministes** (§3.4), le canary teste surtout la **stabilité du harness/scoring**, pas l'aléa du modèle.
- Pour les volets jugés par le modèle (ex. qualité d'une SPEC générée), exécuter K fois et mesurer la dispersion ; comparer à la bande ±8-14 %.
- `lib/canary.js` : exécute, agrège, produit `.aiad/metrics/canary/<date>.json` + verdict global PASS/DRIFT.

### 4.2 Séparation qualité modèle / fiabilité serving
- Deux métriques distinctes dans le rapport :
  - **stabilité déterministe** (les verdicts CLI doivent être 100 % reproductibles — tout écart = bug code, pas modèle),
  - **dispersion modèle** (volets génératifs — bande de tolérance).
- Lien `/aiad retro` : ne déclencher une analyse de régression que si DRIFT hors-bande.

### 4.3 Pin des snapshots
- Documenter et figer dans `config.yml` : `model: claude-opus-4-8`, `effort`, `claude_code_version` de référence.
- Le canary enregistre le snapshot utilisé → comparaisons à modèle constant.

### 4.4 Alignement des références (chasse documentaire)
- Remplacer toutes les occurrences obsolètes : `Opus 4.7` → `Opus 4.8`, `1M` (selon spec 4.8), effort `High` → `xhigh`/`max` là où pertinent, `v2.x` → `v2.1.168`.
- Cibles : `CLAUDE.md`, `AGENTS.md`, `SDDMode.md`, `GUIDE.md`, `frameworkAIAD.md`, skills `context-budget`. Via `emit-rules` quand généré.

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-canary-suite.
2. Constituer `.aiad/canary/cases/` (5-10 cas représentatifs figés).
3. Implémenter `lib/canary.js` (run + agrégation + bande de tolérance) + dispatch CLI `canary`.
4. Brancher sur CI (workflow quotidien `.github/workflows/canary.yml`, cron).
5. Pin des snapshots dans `config.yml`.
6. Chasse d'alignement des références modèles (grep + remplacement) via `emit-rules`.
7. Relier `/aiad retro` au verdict canary.

## 7. Critères d'acceptation (EARS)

- **the system SHALL** rejouer quotidiennement la canary suite et produire un rapport daté.
- **the system SHALL** rapporter 100 % de reproductibilité sur les verdicts déterministes ; tout écart **SHALL** être traité comme un bug code.
- **IF** un volet génératif dévie au-delà de la bande ±14 %, **the system SHALL** marquer DRIFT (régression à investiguer).
- **the system SHALL** référencer Opus 4.8 / effort `xhigh`/`max` / Claude Code v2.1.168 dans la documentation.

## 8. Tests & vérification

- Test `lib/canary.js` : verdict déterministe instable → échec (bug détecté).
- Test bande de tolérance : dispersion < 14 % → PASS ; > 14 % → DRIFT.
- Test d'alignement : `grep -r "Opus 4.7"` ne renvoie plus rien dans les fichiers cibles.

## 9. Risques, dépendances, rollback

- **Risque** : coût des runs génératifs quotidiens → limiter le nombre de cas + cadence (cron nocturne, budget `--max-budget-usd`).
- **Dépendance** : §3.4 (verdicts déterministes = base du canary).
- **Rollback** : canary = observabilité additive, désactivable (workflow off).
