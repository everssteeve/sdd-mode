# Plan — §3.3 Drift Lock → hook `Stop` / `PostToolBatch`

> **Recommandation** : §3.3 de `analyse-claude-code-best-practice.md`
> **Priorité** : P0 · Impact 🔴 Critique · Effort Faible
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

Le Drift Lock (« SPEC + code dans la même PR ») est aujourd'hui une **règle absolue documentaire** + un `pre-commit.sh`. L'analyse (§6.2) veut le rendre **mécaniquement impossible à violer** au moment de la clôture d'une tâche/session : un hook `Stop`/`PostToolBatch` de type `agent` refuse la clôture si `aiad-sdd trace --fail-on-gap` échoue ou si une SPEC n'est pas synchronisée avec le code modifié. « Merger sans Drift Check » devient impossible, pas seulement déconseillé.

## 2. État actuel SDD Mode

| Élément | Fichier réel | Nature |
|---------|--------------|--------|
| Détection de drift | `lib/sdd-trace.js` (`trace`, `--fail-on-gap`), `templates/.claude/skills/drift-detection/SKILL.md` | CLI déterministe ✅ |
| Traçabilité | `lib/sdd-trace.js` + `lib/trace-worker.js` + `lib/trace-cache.js`, annotations `@intent/@spec/@verified-by` | machine-vérifiable ✅ |
| CI | `.github/workflows/sdd-trace.yml` (cité dans CLAUDE.md) | bloque en PR ✅ |
| Hook fin de session | **absent** dans `templates/.claude/settings.json` | ❌ |

**Constat** : tout le calcul déterministe existe déjà (`trace --fail-on-gap`). Il manque uniquement le **branchement harness** à la clôture.

## 3. Cible enforced

Hook `Stop` (et/ou `PostToolBatch` après une rafale d'`Edit`/`Write`) qui :
- exécute `aiad-sdd trace --fail-on-gap --json`,
- si gap bloquant → `decision: "block"` + `asyncRewake` (réveille le modèle avec la liste des annotations manquantes), exit 2,
- sinon laisse la session se clôturer.

## 4. Conception détaillée

### 4.1 Hook `Stop`
```jsonc
{
  "hooks": {
    "Stop": [
      {
        "matcher": "*",
        "hooks": [{
          "type": "command",
          "command": "node ${CLAUDE_PROJECT_DIR}/bin/aiad-sdd.js trace --fail-on-gap --json --quiet",
          "statusMessage": "Drift Lock — vérification traçabilité…",
          "asyncRewake": true,
          "timeout": 180
        }]
      }
    ]
  }
}
```
- Exit 2 + `asyncRewake` ⇒ le modèle est relancé avec le rapport de gap → il pose les annotations `@spec` manquantes ou met à jour la SPEC, puis re-tente la clôture (boucle auto-corrective, façon « Stop : relancer tant que… »).

### 4.2 `PostToolBatch` (optionnel, anticipation)
Après chaque batch d'édition de code applicatif, lancer un `trace` *incrémental* (limité aux fichiers du batch via `lib/trace-cache.js`) pour signaler le drift **tôt**, sans attendre la clôture. Mode `warn` (non bloquant) pour ne pas casser le flow d'édition.

### 4.3 Réutilisation du cache
`lib/trace-cache.js` existe déjà : le hook doit l'exploiter pour rester rapide (< quelques secondes), condition d'acceptabilité d'un hook `Stop`.

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-drift-lock-hook.
2. Garantir un flag `--quiet`/`--json` propre sur `aiad-sdd trace` (sortie machine, exit 0/1/2) — recoupe §3.4.
3. Ajouter le hook `Stop` dans `templates/.claude/settings.json`.
4. (Option) Ajouter `PostToolBatch` en mode `warn` derrière un toggle `hooks-config.json`.
5. Implémenter le mode `asyncRewake` : message de rewake = liste des gaps formatée pour action.
6. Logger dans `.aiad/metrics/hook-runs.jsonl`.

## 6. Fichiers touchés / créés

- **Modifiés** : `lib/sdd-trace.js` (flags `--quiet`, format rewake-friendly), `templates/.claude/settings.json`, `lib/hooks.js`, doc Drift Lock dans `emit-rules`.
- **Créés** : aucun fichier majeur (réutilise l'existant) — éventuellement `lib/drift-lock-hook.js` mince wrapper.

## 7. Critères d'acceptation (EARS)

- **WHEN** une session/agent tente de se clôturer (`Stop`) après modification de code applicatif, **the system SHALL** exécuter `trace --fail-on-gap`.
- **IF** un gap de traçabilité bloquant existe, **the system SHALL** empêcher la clôture (exit 2) et réveiller le modèle avec la liste des annotations manquantes.
- **WHILE** le gap n'est pas résolu, **the system SHALL** rejeter toute nouvelle tentative de clôture.
- **the system SHALL** réutiliser `trace-cache.js` pour rester sous un budget temps acceptable.

## 8. Tests & vérification

- Test : modif de code sans `@spec` → hook `Stop` bloque + message de gap.
- Test : ajout de l'annotation → clôture autorisée.
- Test perf : hook < N s sur un repo moyen grâce au cache.
- Test `PostToolBatch` warn : signal émis sans bloquer.

## 9. Risques, dépendances, rollback

- **Risque** : hook `Stop` lent = friction majeure → cache obligatoire + timeout + dégradation en `warn` si timeout.
- **Risque** : boucle de rewake infinie si l'agent n'arrive pas à corriger → plafonner les rewakes (compteur), puis basculer en JNSP.
- **Dépendance** : §3.4 (sortie JSON/exit codes de `trace`).
- **Rollback** : toggle `disableStopHook`.

## 10. Effort & découpage en SPECs

- SPEC unique, **Effort Faible** (le moteur `trace` existe). Le seul vrai travail neuf = le format `asyncRewake` et le branchement settings. Excellent second hook à livrer après §3.2 (même pattern).
