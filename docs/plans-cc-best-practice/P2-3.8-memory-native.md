# Plan — §3.8 Memory native (agent `memory:`)

> **Recommandation** : §3.8 de `analyse-claude-code-best-practice.md`
> **Priorité** : P2 · Impact 🟡 Moyen · Effort Moyen
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

Les Lessons Learned / Human Learnings de `AGENT-GUIDE.md` et les traces `.aiad/facts/` sont aujourd'hui un markdown que **le modèle peut ignorer** et que rien ne curate. L'analyse (§2.2 « Agent memory ») montre la primitive native : `memory:` frontmatter → store markdown persistant **par agent**, auto-curaté (>200 lignes ⇒ éclatement en fichiers par thème), avec `MEMORY.md` injecté au démarrage. Deux principes clés (§2.4) : mémoire **from logs, pas from one transcript** (« make the button pink » ≠ « tous les boutons roses ») et **anti dock rot** (ne pas garder les artefacts livrés indéfiniment).

## 2. État actuel SDD Mode

| Élément | Fichier réel |
|---------|--------------|
| Contexte permanent + Lessons | `.aiad/AGENT-GUIDE.md` |
| Faits / écarts | `.aiad/facts/` (créé par `/sdd fact`) |
| Réflexion | `lib/reflect.js`, `templates/.claude/aiad/retro.md` |
| Capitalisation feedback | `lib/feedback.js` |

**Manque** : pas de store `memory:` natif, pas d'auto-curation, pas de cycle d'archivage.

## 3. Cible

1. Migrer **Lessons Learned** (AGENT-GUIDE) + `.aiad/facts/` vers un store `memory: project` (versionné + partagé, auto-curaté < 200 lignes).
2. Alimenter la mémoire **from logs** (agrégation de plusieurs sessions/facts) plutôt qu'un transcript isolé.
3. Ajouter un **cycle de clôture/archivage** des artefacts livrés (anti dock rot).

## 4. Conception détaillée

### 4.1 Store `memory: project`
- Les subagents de gouvernance (§3.1) déclarent `memory: project` → `.claude/agent-memory/<agent>/MEMORY.md` (versionné).
- Un agent « curateur » SDD (ou la commande `/aiad retro`) écrit les Lessons dans ce store au format `MEMORY.md` + fichiers thématiques.
- `AGENT-GUIDE.md` garde le **contexte permanent stable** ; les **Lessons volatiles** migrent vers `memory:` (mieux curaté).

### 4.2 Promotion fact → memory (from logs)
- `lib/reflect.js` (ou nouveau `lib/memory.js`) agrège les `.aiad/facts/*` + `hook-runs.jsonl` : ne promeut en mémoire qu'un pattern **récurrent** (seuil d'occurrences), pas un incident unique.
- Commande `aiad-sdd memory promote` : scanne les logs, propose des entrées de mémoire (validation humaine = paternité).

### 4.3 Auto-curation
- Respecter la mécanique native : si `MEMORY.md` > 200 lignes, éclater par thème (`memory/<theme>.md`) + index. `lib/memory.js` peut assister/vérifier ce découpage.

### 4.4 Cycle anti dock rot
- Commande `aiad-sdd archive --delivered` (lib `lib/archive.js` existe déjà) : déplace les SPECs/Intents/Research **livrés et clos** vers `.aiad/_archive/` ; ne garde en contexte chaud que l'actif.
- Hook `SessionStart`/`InstructionsLoaded` : ne charger que la mémoire active + les 200 premières lignes de `MEMORY.md`.

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-memory-native.
2. Définir le format de store mémoire (réutiliser convention `MEMORY.md` native).
3. Implémenter `lib/memory.js` : `promote` (from logs, seuil), `curate` (split >200 lignes), `archive` (délégué à `lib/archive.js`).
4. Déclarer `memory: project` sur les agents de gouvernance (dépend §3.1).
5. Migrer les Lessons existantes de `AGENT-GUIDE.md` vers le store (one-shot, validé humain).
6. Brancher `/aiad retro` et `/sdd fact` sur `lib/memory.js`.
7. Ajouter le cycle d'archivage (commande + rappel dans `/aiad health`).

## 6. Fichiers touchés / créés

- **Créés** : `lib/memory.js`, store `.claude/agent-memory/*/MEMORY.md` (généré), tests.
- **Modifiés** : `.aiad/AGENT-GUIDE.md` (allégé), `lib/reflect.js`, `lib/archive.js`, `templates/.claude/aiad/retro.md`, `templates/.claude/sdd/fact.md`, `bin/aiad-sdd.js`, `templates/.claude/agents/*` (frontmatter `memory:`).

## 7. Critères d'acceptation (EARS)

- **the system SHALL** persister les Lessons Learned dans un store `memory: project` versionné.
- **WHEN** un pattern apparaît dans ≥ N facts/logs, **the system SHALL** le proposer à la promotion en mémoire (jamais auto-promu sans validation humaine).
- **IF** `MEMORY.md` dépasse 200 lignes, **the system SHALL** l'éclater en fichiers thématiques.
- **WHEN** une SPEC/Intent/Research est close et livrée, **the system SHALL** permettre son archivage hors contexte chaud.

## 8. Tests & vérification

- Test `promote` : 1 occurrence → non promu ; N occurrences → proposé.
- Test `curate` : store > 200 lignes → split correct + index.
- Test archivage : artefact livré déplacé, contexte chaud réduit.
- Test paternité : aucune promotion sans flag de validation humaine.

## 9. Risques, dépendances, rollback

- **Risque** : perte d'historique lors de la migration → archivage, pas suppression.
- **Risque** : sur-mémorisation (bruit) → seuil d'occurrences + validation humaine.
- **Dépendance** : §3.1 (agents avec `memory:`), §3.10 (canary distingue régression réelle du bruit avant de mémoriser).
- **Rollback** : conserver `AGENT-GUIDE.md` comme source ; la mémoire native est additive.

## 10. Effort & découpage en SPECs

- SPEC-B : cycle d'archivage anti dock rot. *(Faible)*

**Effort global** : Moyen.
