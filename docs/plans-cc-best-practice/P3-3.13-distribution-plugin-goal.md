# Plan — §3.13 Distribution plugin/marketplace + boucles `/goal`

> **Recommandation** : §3.13 de `analyse-claude-code-best-practice.md`
> **Priorité** : P3 · Impact 🟢 Confort · Effort Variable
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

Trois leviers de diffusion/automatisation (§2.4, §2.5) :
1. **Packager SDD comme plugin/marketplace** (modèle `anthropics/skills`) → distribution à l'échelle (Thariq : « distribution via marketplace de plugins »).
2. **`/goal`** (boucle évaluée par Haiku jusqu'à condition) → « itérer jusqu'à SQS ≥ 4/5 » devient une primitive native (Ralph Wiggum loop).
3. **Pattern config plate de toggles** (`hooks-config.json` + `.local.json`) → activer/désactiver les mécanismes SDD par environnement.

SDD est déjà un package npm (`aiad-sdd`) ; il s'agit de l'exposer aussi comme **plugin Claude Code** et d'exploiter les boucles natives.

## 3. Cible

1. **Plugin marketplace** : manifeste de plugin Claude Code exposant commandes + skills + agents SDD, installable via marketplace.
2. **Boucles `/goal`** : intégrer/documenter `/goal "SQS ≥ 4/5"` autour de `/sdd gate` et `/goal "0 gap"` autour de `/sdd trace`.
3. **Config plate de toggles** : `hooks-config.json` (booléens `disable<Event>Hook`) + `.local.json` gitignored, pour activer/désactiver par environnement les hooks P0.

## 4. Conception détaillée

### 4.1 Plugin/marketplace
- Créer un manifeste plugin (format Claude Code plugins, cf. `lib/plugins.js` existant) listant : commandes `templates/.claude/{sdd,aiad}/`, skills, agents (§3.1), settings.
- Publier sur un marketplace (repo `everssteeve/sdd-mode` comme registre) ; doc d'installation `/plugin install aiad-sdd`.
- Réutiliser `lib/governance-marketplace.js` / `lib/marketplace.js` (packs de gouvernance déjà distribuables).

### 4.2 Boucles `/goal`
- Documenter et fournir des recettes : `/goal "aiad-sdd gate --json renvoie verdict=PASS"` → la condition est évaluée par Haiku sur la **sortie déterministe** (§3.4) — la condition est donc machine-vérifiable, pas un jugement flou.
- Idem `/goal "aiad-sdd trace --fail-on-gap exit 0"`.
- Garde-fou : plafond d'itérations + bascule JNSP si non-convergence (cohérent §3.6 rewake cap).

### 4.3 Config plate de toggles
- `hooks-config.json` (à la racine `.aiad/` ou `.claude/`) : `{ "disablePreToolUseHook": false, "disableStopHook": false, ... }` + `hooks-config.local.json` (gitignored, override env).
- Les hooks P0 (§3.1/3.2/3.3) lisent ce fichier → activation/désactivation par environnement sans toucher au code (pattern réel du repo de référence, §2.5).

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-distribution-plugin + SPEC-NNN-hooks-toggles.
2. Créer le manifeste plugin + brancher `lib/plugins.js`.
3. Rédiger les recettes `/goal` (doc + exemples) autour de gate/trace.
4. Implémenter le chargement `hooks-config.json` + `.local.json` dans les hooks P0 (lib partagée `lib/hooks-config.js`).
5. Documenter l'installation marketplace dans README + `docs/marketplace-packs.md`.
6. Ajouter au `.gitignore` le `hooks-config.local.json`.

## 6. Fichiers touchés / créés

- **Créés** : manifeste plugin, `lib/hooks-config.js`, `hooks-config.json` (+ `.local.json` exemple), recettes `/goal` (doc).
- **Modifiés** : `lib/plugins.js`, `lib/marketplace.js`, `.aiad/hooks/*` (lecture toggles), `.gitignore`, README, `docs/marketplace-packs.md`.

## 7. Critères d'acceptation (EARS)

- **the system SHALL** être installable comme plugin Claude Code via marketplace (commandes + skills + agents).
- **the system SHALL** fournir des recettes `/goal` dont la condition est évaluée sur une sortie CLI déterministe.
- **WHERE** `hooks-config.json` désactive un hook, **the system SHALL** ne pas l'exécuter.
- **the system SHALL** permettre un override local gitignored (`hooks-config.local.json`).

## 8. Tests & vérification

- Test : `disableStopHook: true` → hook `Stop` non exécuté.
- Test : override local prioritaire sur le partagé.
- Test plugin : install dans un projet vierge → commandes disponibles.
- Test `/goal` : convergence sur PASS, plafond respecté.

## 9. Risques, dépendances, rollback

- **Risque** : un toggle qui désactive un veto enforced → marquer les hooks de gouvernance **non désactivables** côté managed (§3.1, `allowManagedHooksOnly`).
- **Dépendance** : §3.1/3.2/3.3 (hooks à toggler), §3.4 (conditions `/goal` déterministes).
- **Rollback** : la distribution npm reste la voie par défaut ; le plugin est additif.
