# Plan — §3.9 Cycle SDD comme graphe de Tasks

> **Recommandation** : §3.9 de `analyse-claude-code-best-practice.md`
> **Priorité** : P2 · Impact 🟡 Moyen · Effort Moyen
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

« Ne jamais sauter d'étape » du cycle SDD est aujourd'hui une **règle textuelle**. L'analyse (§2.2, §6.2) propose de la rendre **exécutoire** via les **Tasks** natives (v2.1.16, remplacent TodoWrite) : filesystem `~/.claude/tasks/`, **auditable + versionnable + crash-recoverable**, avec un **graphe de dépendances** (`addBlockedBy`/`addBlocks`) et un support multi-session (`CLAUDE_CODE_TASK_LIST_ID`). On modélise `Intent → Research → SPEC → Gate → Exec → Validate → Drift Lock` comme un graphe : une étape **bloquée** par la précédente ne peut pas démarrer ; `crash-recoverable` répond nativement à `/sdd resume`.

## 2. État actuel SDD Mode

| Élément | Fichier réel |
|---------|--------------|
| Étapes du cycle | `templates/.claude/sdd/*.md` (intent, spec, gate, exec, validate, drift-check, trace) |
| Reprise | `templates/.claude/sdd/resume.md` |
| Statut projet | `lib/status.js`, `templates/.claude/aiad/status.md` |
| Santé artefacts | `lib/doctor.js`, `templates/.claude/aiad/health.md` |

**Manque** : aucune matérialisation du cycle en graphe de tâches à dépendances ; l'ordre repose sur la discipline du modèle.

## 3. Cible

Chaque lancement d'un cycle (par Intent/Research) crée un **graphe de Tasks** :
```
INTENT ──blocks──▶ RESEARCH ──blocks──▶ SPEC ──blocks──▶ GATE ──blocks──▶ EXEC ──blocks──▶ VALIDATE ──blocks──▶ DRIFT-LOCK
```
- Une tâche `blockedBy` non complétée empêche le démarrage de la suivante.
- `CLAUDE_CODE_TASK_LIST_ID` rattache le graphe à l'Intent → multi-session + crash recovery → `/sdd resume` lit l'état réel.

## 4. Conception détaillée

### 4.1 Génération du graphe
- `/sdd intent` (ou `/sdd research`) crée la liste de tâches du cycle via l'API Tasks (`TaskCreate` + `addBlockedBy`).
- ID de liste = dérivé de `INTENT-NNN` → un graphe par intention.
- `lib/cycle-graph.js` : construit/lit/met à jour le graphe (wrapper autour des Tasks).

### 4.2 Transitions pilotées par verdicts
- Une tâche passe `completed` **uniquement** quand le verdict déterministe associé est PASS (exit 0) — lien §3.4.
  - `GATE` → completed si `aiad-sdd gate` = PASS.
  - `VALIDATE` → completed si `validate` = PASS.
  - `DRIFT-LOCK` → completed si `trace --fail-on-gap` = PASS.
- JNSP (exit 2) → tâche en `blocked` avec note JNSP.

### 4.3 Resume natif
- `/sdd resume` lit le graphe : reprend à la première tâche non-`completed` non-`blocked`, ou signale la tâche `blocked` + sa raison.
- `crash-recoverable` : si la session meurt en EXEC, le graphe sur filesystem garde l'état.

### 4.4 Statut visuel
- `lib/status.js` affiche le graphe (étapes `[x]/[~]/[ ]/[!]`) — cohérent avec le statut des tranches de §3.6.

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-cycle-tasks-graph.
2. Implémenter `lib/cycle-graph.js` (création/lecture/maj du graphe via Tasks API).
3. Brancher la création du graphe sur `/sdd intent` (ou `/sdd research`).
4. Câbler les transitions sur les exit codes des verdicts (§3.4).
5. Réécrire `/sdd resume` pour lire le graphe.
6. Étendre `lib/status.js` / `/aiad status` à l'affichage du graphe.
7. Documenter le mapping étape↔task.

## 6. Fichiers touchés / créés

- **Créés** : `lib/cycle-graph.js`, tests.
- **Modifiés** : `templates/.claude/sdd/{intent,research,resume}.md`, `lib/status.js`, `templates/.claude/aiad/status.md`, `bin/aiad-sdd.js`, doc cycle.

## 7. Critères d'acceptation (EARS)

- **WHEN** un Intent démarre un cycle, **the system SHALL** créer un graphe de Tasks reliant les étapes par `blockedBy`.
- **IF** l'étape N-1 n'est pas `completed`, **the system SHALL** empêcher le démarrage de l'étape N.
- **WHEN** un verdict déterministe d'étape est PASS, **the system SHALL** marquer la tâche correspondante `completed`.
- **WHEN** une session reprend, **the system SHALL** restaurer l'état du cycle depuis le graphe persistant (crash-recoverable).

## 8. Tests & vérification

- Test : tenter EXEC alors que GATE n'est pas PASS → bloqué.
- Test : verdict PASS → transition de tâche.
- Test crash : interruption en EXEC → `/sdd resume` reprend au bon nœud.
- Test multi-session : même `TASK_LIST_ID` partagé.

## 9. Risques, dépendances, rollback

- **Risque** : dépendance à une API harness (Tasks) qui peut évoluer → encapsuler dans `lib/cycle-graph.js` (couche d'abstraction) + fallback fichier `.aiad/cycle/<intent>.json`.
- **Dépendance** : §3.4 (exit codes pilotent les transitions), §3.5 (étape Research dans le graphe).
- **Rollback** : le graphe est un *miroir* exécutoire ; sans lui, le cycle reste fonctionnel en mode advisory.

## 10. Effort & découpage en SPECs

- SPEC unique avec fallback fichier. **Effort Moyen** (l'essentiel est le wrapper + les transitions).
