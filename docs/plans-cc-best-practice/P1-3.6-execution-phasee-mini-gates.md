# Plan — §3.6 Exécution phasée + mini-gates + 3e verdict `CONDITIONAL PASS`

> **Recommandation** : §3.6 de `analyse-claude-code-best-practice.md`
> **Priorité** : P1 · Impact 🟠 Fort · Effort Moyen
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

`/sdd exec` lance l'agent avec **une Execution Gate unique** en amont, puis laisse filer. L'analyse (§2.3, §2.4) montre deux faiblesses : (a) le modèle « code horizontalement » — 1200 lignes avant le moindre test (Dex) ; (b) un verdict binaire PASS/FAIL est trop pauvre — RPI utilise un 3e état `CONDITIONAL PASS` et un **scoring 0-10 par dimension avec evidence** (constitutional-validator). On découpe donc l'exécution en **tranches verticales testables**, avec **mini-gates répétés** + **statut visuel**, on ajoute **`CONDITIONAL PASS`**, et on emprunte le **scoring dimensionnel + evidence** — tout en gardant `UNKNOWN = VETO`.

## 2. État actuel SDD Mode

| Élément | Fichier réel |
|---------|--------------|
| Exécution | `templates/.claude/sdd/exec.md` |
| Gate SQS (5 critères) | `lib/score.js` (`scorerArtefact`, `VERDICTS`), `templates/.claude/skills/sqs-scoring/SKILL.md` |
| Reprise de session | `templates/.claude/sdd/resume.md` |
| Découpe SPEC | `templates/.claude/sdd/split.md`, `lib/refactor-spec.js` |

**Manque** : exécution en une passe, gate unique, verdicts binaires, pas de statut de progression machine.

## 3. Cible

1. **Plan d'exécution phasé** : `/sdd exec` génère N tranches verticales testables (chaque tranche livre un incrément + ses tests `@verified-by`).
2. **Mini-gate par tranche** : à la fin de chaque tranche, mini-validation (tests verts + drift-check incrémental) avec statut visuel `[ ] [~] [x] [!] [-]`.
3. **3e verdict `CONDITIONAL PASS`** ajouté au moteur de verdict.
4. **Scoring 0-10 par dimension + evidence** optionnel pour les validations riches.

## 4. Conception détaillée

### 4.1 Plan d'exécution phasé
- `/sdd exec` produit un artefact `.aiad/exec/EXEC-<spec>-plan.md` listant les phases (tranches verticales), chacune avec : objectif, fichiers, tests, critère de done.
- Inspiré `/rpi:implement` + « tranches verticales testables » (Dex) : interdiction de coder horizontalement.
- Statut machine dans le plan : `- [ ]` à faire, `[~]` en cours, `[x]` validé, `[!]` bloqué, `[-]` hors-scope.

### 4.2 Mini-gate par tranche
- Après chaque tranche : `aiad-sdd mini-gate --phase N` → lance les tests de la tranche + `trace` incrémental (cache) → verdict.
- Verdicts : `PASS | CONDITIONAL PASS | FAIL | JNSP`.
  - `CONDITIONAL PASS` = tranche acceptable avec dette explicitée (conditions listées, à lever avant la gate finale).
- Lien enforced (P0) : hook `PostToolBatch` warn signalant qu'une tranche est terminée sans tests verts.

### 4.3 3e verdict dans le moteur
- Étendre `lib/score.js` `VERDICTS` : ajouter `CONDITIONAL` (entre PASS et FAIL).
- Exit codes : PASS=0, CONDITIONAL=0 **avec warning** (ou code dédié documenté), FAIL=1, JNSP=2. Décision à arbitrer (cohérence §3.4) — proposer CONDITIONAL=0 + champ `conditions[]` non vide dans le JSON.
- `UNKNOWN = VETO` reste : un critère non scorable force JNSP, jamais CONDITIONAL.

### 4.4 Scoring dimensionnel + evidence
- Option `--dimensions` sur `gate`/`validate` : chaque critère SQS scoré 0-10 (au lieu de 0-5) avec **evidence citée** (fichier:ligne), à l'image du constitutional-validator. Détecte anti-patterns nommés (scope creep, over-engineering).
- Rétro-compat : le scoring 0-5 reste le défaut ; 0-10 = mode riche opt-in.

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-exec-phasee + SPEC-NNN-conditional-verdict.
2. Étendre `lib/score.js` : verdict `CONDITIONAL` + champ `conditions[]` + (option) scoring 0-10 + evidence.
3. Créer `lib/mini-gate.js` (tests de tranche + trace incrémental) ; dispatch CLI `mini-gate`.
4. Faire générer le plan phasé par `templates/.claude/sdd/exec.md` (+ artefact `.aiad/exec/`).
5. Introduire le statut visuel `[ ][~][x][!][-]` et son parsing (`lib/exec-status.js`).
6. Brancher `/sdd resume` sur le plan phasé (reprise à la tranche `[~]`/`[!]`).
7. Mettre à jour `sqs-scoring/SKILL.md` (3e verdict + dimensions).

## 7. Critères d'acceptation (EARS)

- **WHEN** `/sdd exec` démarre, **the system SHALL** produire un plan en tranches verticales testables, chacune avec ses tests.
- **WHEN** une tranche se termine, **the system SHALL** exécuter un mini-gate (tests + drift incrémental) renvoyant `PASS | CONDITIONAL PASS | FAIL | JNSP`.
- **the system SHALL** exposer un verdict `CONDITIONAL PASS` portant une liste de conditions non vide.
- **IF** un critère est non scorable, **the system SHALL** renvoyer JNSP (jamais CONDITIONAL).
- **the system SHALL** refléter l'avancement par les marqueurs `[ ] [~] [x] [!] [-]`.

## 8. Tests & vérification

- Test `lib/score.js` : entrée → CONDITIONAL avec conditions ; critère manquant → JNSP.
- Test `lib/mini-gate.js` : tranche sans tests verts → FAIL.
- Test reprise : `/sdd resume` repart à la tranche `[~]`.
- Test parsing statut.

## 9. Risques, dépendances, rollback

- **Risque** : sur-process pour petites SPECs → nombre de tranches proportionné (1 tranche autorisée).
- **Risque** : ambiguïté sur l'exit code de CONDITIONAL → trancher avec §3.4 et documenter une fois pour toutes.
- **Dépendance** : §3.4 (moteur de verdict), §3.3 (drift incrémental).
- **Rollback** : mode phasé activable via `config.yml` (`exec.phased: false` ⇒ comportement v1.17 actuel).
