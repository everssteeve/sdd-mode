# Plan — §3.4 Verdicts machine-vérifiables (« computation off-context »)

> **Recommandation** : §3.4 de `analyse-claude-code-best-practice.md`
> **Priorité** : P0 · Impact 🔴 Critique · Effort Faible
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

Finding critique de l'analyse (§2.2 « SDK vs CLI » + §3.4) : **aucun déterminisme LLM n'est garanti** (pas de `seed`, variance floating-point + routing MoE + batching, dérive ±8-14 % §2.2). Donc **le verdict final d'une gate ne doit jamais venir du jugement libre du modèle**, mais d'un **script CLI déterministe** dont la sortie est validée par schéma et signalée par exit code. C'est le principe « computation off-context » du Programmatic Tool Calling : calcul hors-modèle, seul le verdict entre en contexte. SDD le fait **déjà** pour `trace` ; il faut l'étendre à `gate`, `validate`, `security`.

## 2. État actuel SDD Mode

|----------|-----|:-----------:|:----------------:|:------:|
| `trace` | `lib/sdd-trace.js` | partielle (`trace.json`) | partielle | ❓ |

- `lib/cli-schema.js` + `.aiad/schema/cli-openapi.yaml` existent déjà → socle de schémas exploitable.

## 3. Cible enforced

Chaque commande de verdict expose :
- `--print --output-format json`
- `--json-schema <chemin>` (sortie validée contre un JSON Schema versionné)
- **exit codes stricts** : `0` = PASS, `1` = FAIL, `2` = JNSP/UNKNOWN (décision humaine requise, **pas** une erreur).

## 4. Conception détaillée

### 4.1 Schémas de verdict versionnés
Créer `.aiad/schema/verdicts/` :
- `gate.schema.json` — `{ verdict: "PASS|FAIL|INCONNUE", sqs: { criteres: [{id,score,evidence}], total }, remediation: [...] }`
- `validate.schema.json`, `security.schema.json`, `trace.schema.json`.

Source unique pour le code (`lib/score.js`) **et** les hooks **et** la CI.

### 4.2 Contrat de sortie commun
Module partagé `lib/verdict.js` :
```js
export function emitVerdict({ verdict, payload, schemaPath, json }) {
  // 1. valide payload contre schemaPath (ajv ou validateur maison cli-schema.js)
  // 2. si json → stdout = JSON.stringify(payload)
  // 3. exit code = { PASS:0, FAIL:1, JNSP:2 }[verdict]
}
```
Toutes les commandes de verdict passent par `emitVerdict` → garantie d'uniformité.

### 4.3 Branchement existant
- `lib/score.js` (`VERDICTS`) : mapper sur exit codes via `verdict.js`.
- `bin/aiad-sdd.js` : ajouter `--output-format`, `--json-schema` au `parseArgs` global.
- Réutiliser `lib/cli-schema.js` pour la validation.

### 4.4 Conséquence harness
Les hooks de §3.1/§3.2/§3.3 consomment ces sorties JSON + exit codes → cohérence totale : le verdict affiché à l'humain, injecté au modèle, et lu par le hook sont **le même artefact déterministe**.

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-verdicts-machine (EARS — contrat d'API public).
2. Définir les 4 JSON Schemas dans `.aiad/schema/verdicts/`.
3. Implémenter `lib/verdict.js` (`emitVerdict`) + tests.
4. Ajouter les flags `--output-format json|stream-json` et `--json-schema` au dispatch `bin/aiad-sdd.js`.
5. Câbler `gate`/`score`, `validate`, `security`, `trace` sur `emitVerdict`.
6. Garantir exit codes 0/1/2 uniformes (audit de chaque `exit()` dans les libs concernées).
7. Mettre à jour la doc « Exit codes CLI » de `CLAUDE.md` (déjà tabulée) avec exemples `--json`.
8. Mettre à jour `.aiad/schema/cli-openapi.yaml`.

## 7. Critères d'acceptation (EARS)

- **WHEN** une commande de verdict est appelée avec `--output-format json`, **the system SHALL** émettre un JSON conforme au schéma déclaré.
- **IF** `--json-schema` est fourni et la sortie ne valide pas, **the system SHALL** échouer (exit 1) avant émission.
- **the system SHALL** retourner exit `0` pour PASS, `1` pour FAIL, `2` pour JNSP/UNKNOWN, pour `gate`, `validate`, `security`, `trace`.
- **the system SHALL** produire un verdict **identique** quel que soit le canal (humain, JSON, hook).

## 8. Tests & vérification

- Tests de contrat : chaque commande × {PASS, FAIL, JNSP} → exit code + JSON valide attendus.
- Test schéma : payload malformé rejeté par `--json-schema`.
- Test de cohérence : sortie humaine et JSON portent le même verdict.
- Snapshot tests des schémas (régression de format).

## 9. Risques, dépendances, rollback

- **Risque** : casser des intégrations qui lisent l'ancienne sortie texte → versionner (`--output-format json` opt-in, texte = défaut inchangé).
- **Dépendance** : **prérequis** de §3.1, §3.2, §3.3 (les hooks lisent ces sorties). À livrer **en premier** du lot P0.
- **Rollback** : flags opt-in ⇒ retrait sans impact sur l'usage texte existant.

## 10. Effort & découpage en SPECs

- SPEC unique, **Effort Faible** mais **fondationnel** : à prioriser avant les 3 hooks P0 (3.1/3.2/3.3 en dépendent). Le socle (`cli-schema.js`, `cli-openapi.yaml`, `score.js VERDICTS`) existe déjà.
