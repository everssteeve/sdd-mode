# Plan — §3.7 Budget d'instructions : push → pull (progressive disclosure)

> **Recommandation** : §3.7 de `analyse-claude-code-best-practice.md`
> **Priorité** : P1 · Impact 🟠 Fort · Effort Faible
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

Dex (§2.4) : **~150-200 instructions max** avant que l'adhérence s'effondre. Aujourd'hui SDD **pousse** en permanence : `CLAUDE.md` (16 KB) + `AGENTS.md` + 4 AGENT-GUIDEs Tier 1 + matrices, tous injectés à froid. Cela dilue l'adhérence (le harness ne peut pas suivre 500 instructions). L'analyse veut passer en **pull** : charger la gouvernance **à la demande** par zone de risque (`paths:`), garder `CLAUDE.md` le plus court possible, et appliquer la **progressive disclosure** sur les skills lourdes. C'est l'application directe de « Sobriété Intentionnelle ».

## 2. État actuel SDD Mode

| Élément | Fichier réel | Charge |
|---------|--------------|--------|
| Instructions principales | `CLAUDE.md` (16 069 o) + `AGENTS.md` (synchro) | injecté à froid |
| Gouvernance | `.aiad/gouvernance/*.md` (4 agents) | référencé en permanence |
| Skills | `templates/.claude/skills/*/SKILL.md` (8) | descriptions en contexte |
| Settings budget | `templates/.claude/settings.json` | pas de `maxSkillDescriptionChars`, ni `AUTOCOMPACT_PCT_OVERRIDE` |

## 3. Cible

1. **`.claude/rules/*.md` avec `paths:`** : la gouvernance RGPD ne se charge que sur fichiers PII, RGAA que sur UI, etc. (lazy-load).
2. **`CLAUDE.md` minimal** : noyau court + pointeurs ; le détail descend en `references/`.
3. **Progressive disclosure** sur les skills lourdes (`SKILL.md` court + `references/*.md` + `examples.md`).
4. **Réglages de budget** explicites : `skillListingBudgetFraction`, `maxSkillDescriptionChars`, `env.CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` (= levier natif du seuil 60-70 %).

## 4. Conception détaillée

### 4.1 Gouvernance en pull (`paths:`)
Créer `templates/.claude/rules/` généré par `emit-rules` :
```yaml
# templates/.claude/rules/rgpd.md
---
paths: ["**/*pii*","**/*user*","**/auth/**","**/*personal*"]
---
<condensé RGPD opérationnel — chargé uniquement sur ces fichiers>
```
Idem `rgaa.md` (`paths: ["**/*.tsx","**/components/**","**/*.vue",...]`), `rgesn.md`, `ai-act.md` (`paths` sur composants ML/LLM). Source unique : `.aiad/gouvernance/`.
⚠️ Rappel analyse (§2.1) : les skills **ne remontent pas** l'arbre ; les `rules` `paths:` sont la primitive correcte pour la gouvernance par zone.

### 4.2 CLAUDE.md minimal
- Réduire `CLAUDE.md` au noyau non-négociable (identité, Human Authorship, JNSP, hiérarchie, pointeurs).
- Déplacer les tableaux exhaustifs de commandes vers `references/commands.md` (chargé à la demande / via `/aiad-help`).
- Cible indicative : < ~250 lignes de contexte permanent (vs 16 KB actuel).

### 4.3 Progressive disclosure des skills
Pour chaque skill lourde (`traceability`, `regulatory-veto`, `sqs-scoring`, `drift-detection`) :
- `SKILL.md` = description courte + table de pointeurs.
- `references/*.md`, `examples.md`, `templates/*` chargés à l'invocation.
- Vérifier que chaque `description` ≤ `maxSkillDescriptionChars` (1536) et que la somme tient sous `skillListingBudgetFraction` (1 %).

### 4.4 Réglages de budget (settings émis)
```jsonc
{
  "env": { "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "65" },   // seuil 60-70 % rendu natif
  "maxSkillDescriptionChars": 1536,
  "skillListingBudgetFraction": 0.01
}
```
+ `context: fork` + agent `Explore` (haiku read-only) pour les skills read-only (drift-detection, scan d'annotations) → calcul hors contexte principal.

### 4.5 Mesure
Étendre `templates/.claude/skills/context-budget/SKILL.md` + `lib/coldstart.js` (`bench`) pour mesurer le **gain réel** de tokens à froid (avant/après push→pull).

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-context-pull.
2. Mesurer la baseline (`aiad-sdd bench`/coldstart) : tokens chargés à froid aujourd'hui.
3. Générer `templates/.claude/rules/{rgpd,rgaa,rgesn,ai-act}.md` avec `paths:` via `emit-rules`.
4. Refondre `CLAUDE.md`/`AGENTS.md` en noyau minimal + `references/`.
5. Restructurer les 4 skills lourdes en progressive disclosure.
6. Ajouter les réglages de budget dans `templates/.claude/settings.json`.
7. Re-mesurer ; documenter le gain dans le CHANGELOG-ARTEFACTS.
8. Étendre `aiad-emit-rules-check.yml` aux nouveaux `rules/`.

## 6. Fichiers touchés / créés

- **Créés** : `templates/.claude/rules/*.md`, `references/*.md` (commands, gouvernance détaillée), `references/`/`examples.md` par skill lourde.
- **Modifiés** : `CLAUDE.md`, `templates/CLAUDE.md`, `AGENTS.md` (via emit), `lib/emit-rules.js`, `templates/.claude/settings.json`, 4 `SKILL.md`, `lib/coldstart.js`, `context-budget/SKILL.md`.

## 7. Critères d'acceptation (EARS)

- **WHERE** un fichier matche le `paths:` d'une rule de gouvernance, **the system SHALL** charger cette rule ; sinon **the system SHALL NOT** la charger.
- **the system SHALL** réduire le contexte permanent chargé à froid d'au moins X % (cible mesurée vs baseline).
- **the system SHALL** garder chaque description de skill ≤ 1536 caractères.
- **the system SHALL** régler `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` à 65 dans les settings émis.
- **the system SHALL** préserver la source unique `.aiad/gouvernance/` (rules générées, non éditées à la main).

## 8. Tests & vérification

- Test `emit-rules` : rules générées conformes à la source (CI).
- Test budget : `bench` montre un gain ≥ cible.
- Test lazy-load : ouvrir un fichier UI charge `rgaa.md`, pas `rgpd.md` (vérif manuelle/headless).
- Test non-régression : aucune règle critique perdue (couverture par les tests de gouvernance existants).

## 9. Risques, dépendances, rollback

- **Risque majeur** : une règle de veto qui ne se charge plus là où elle devrait → la couche **enforced** (§3.1 hooks `deny`/`agent`) reste le vrai garde-fou ; le pull n'allège que l'*advisory*. Les deux sont complémentaires.
- **Risque** : `paths:` mal calibrés (faux négatifs) → tests de couverture des globs + revue humaine.
- **Dépendance** : §3.1 (les vetos enforced compensent l'allègement advisory).
- **Rollback** : restaurer `CLAUDE.md` complet ; les `rules/` sont additives.

## 10. Effort & découpage en SPECs

- SPEC-A : `rules/` `paths:` + réglages settings. *(Faible)*
- SPEC-B : refonte CLAUDE.md minimal + progressive disclosure skills. *(Faible-Moyen)*

**Effort global** : Faible. **À coordonner impérativement avec §3.1** : ne jamais alléger une règle de veto sans qu'elle soit déjà enforced.
