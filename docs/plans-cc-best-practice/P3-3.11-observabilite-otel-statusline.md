# Plan — §3.11 Observabilité native (OpenTelemetry + statusLine)

> **Recommandation** : §3.11 de `analyse-claude-code-best-practice.md`
> **Priorité** : P3 · Impact 🟢 Confort · Effort Variable
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

SDD produit déjà des métriques DORA/Flow, mais via des commandes maison (`/aiad dora`, `/aiad flow`) écrivant dans `.aiad/metrics/`. L'analyse (§2.1, §6.2) propose de **brancher l'observabilité sur les primitives natives** : **OpenTelemetry** (`OTEL_*`) pour exporter DORA/Flow vers n'importe quel backend, une **`statusLine`** live (SPEC active, état Gate, % contexte), la **mesure d'usage des skills** via hook `PreToolUse`, et `attribution.commit` pour imposer le `Co-Authored-By` sans dépendre du modèle.

## 2. État actuel SDD Mode

| Élément | Fichier réel |
|---------|--------------|
| DORA / Flow | `lib/dora-record.js`, `templates/.claude/aiad/{dora,flow}.md`, `docs/dora-format.md`, `docs/flow-format.md` |
| Télémétrie | `lib/telemetry.js` (opt-in/out) |
| Métriques | `.aiad/metrics/*` |
| Settings | `templates/.claude/settings.json` (pas d'`OTEL_*`, pas de `statusLine`, pas d'`attribution`) |

## 3. Cible

1. **Export OTel** : `env.OTEL_*` dans les settings ; les métriques DORA/Flow émises comme métriques OTel.
2. **statusLine** : commande qui affiche SPEC active + état Gate + % contexte + effort.
3. **Mesure d'usage des skills** : hook `PreToolUse` matcher `Skill` → log JSONL.
4. **attribution.commit** : `Co-Authored-By` imposé par settings, pas par le modèle.

## 4. Conception détaillée

### 4.1 OTel
- `templates/.claude/settings.json` : bloc `env` optionnel `OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_METRICS_EXPORTER`, `OTEL_RESOURCE_ATTRIBUTES` (service.name=aiad-sdd).
- `lib/telemetry.js` étendu : émettre les compteurs DORA/Flow au format OTel (opt-in, respecte le consentement existant).

### 4.2 statusLine
- `lib/statusline.js` : lit stdin JSON (`cost.total_cost_usd`, `context_window.used_percentage`, `effort.level`, `github`) + état SDD (SPEC active depuis `.aiad/specs/_index.md`, dernier verdict gate).
- Affichage : `SPEC-042 │ Gate ✅ │ ctx 58% │ effort max`.
- Déclaré via `statusLine.command` dans settings.

### 4.3 Mesure d'usage des skills
- Hook `PreToolUse` matcher `Skill` → append `.aiad/metrics/skill-usage.jsonl` (skill, timestamp, contexte). Permet de savoir **quelles skills servent vraiment** (Sobriété Intentionnelle : retirer les inutilisées).

### 4.4 Attribution
- `settings.attribution.commit` + `attribution.pr` dans les settings émis → `Co-Authored-By` garanti.

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-observabilite-native.
2. Implémenter `lib/statusline.js` + déclaration `statusLine.command`.
3. Étendre `lib/telemetry.js` à l'export OTel (opt-in).
4. Ajouter le hook `PreToolUse` matcher `Skill` (mesure d'usage).
5. Ajouter `attribution` aux settings émis.
6. Documenter dans `docs/telemetrie.md` (existant) le branchement OTel.

## 6. Fichiers touchés / créés

- **Créés** : `lib/statusline.js`, `.aiad/metrics/skill-usage.jsonl` (généré), tests.
- **Modifiés** : `templates/.claude/settings.json`, `lib/telemetry.js`, `docs/telemetrie.md`, `bin/aiad-sdd.js`.

## 7. Critères d'acceptation (EARS)

- **WHERE** OTel est configuré, **the system SHALL** exporter les métriques DORA/Flow au format OTel (sous réserve du consentement télémétrie).
- **the system SHALL** afficher dans la statusLine la SPEC active, l'état de la dernière Gate, le % de contexte et l'effort.
- **WHEN** une skill est invoquée, **the system SHALL** journaliser son usage.
- **the system SHALL** apposer `Co-Authored-By` via `attribution.commit` indépendamment du modèle.

## 8. Tests & vérification

- Test `lib/statusline.js` : stdin JSON → ligne attendue.
- Test OTel : compteurs émis (mock exporter).
- Test mesure skills : invocation → ligne JSONL.

## 9. Risques, dépendances, rollback

- **Risque** : fuite de données via OTel → respecter `lib/telemetry.js` opt-in + `OTEL_RESOURCE_ATTRIBUTES` sans PII.
- **Dépendance** : aucune dure ; complète §3.10 (canary peut consommer ces métriques).
- **Rollback** : tout est opt-in via settings/env.

## 10. Effort & découpage en SPECs

- SPEC-A : statusLine + attribution. *(Faible)*
- SPEC-B : OTel export + mesure skills. *(Moyen)*

**Effort global** : Variable (Faible si on s'en tient à statusLine + attribution).
