# Plan — §3.12 Cross-model auteur/reviewer « additive-only »

> **Recommandation** : §3.12 de `analyse-claude-code-best-practice.md`
> **Priorité** : P3 · Impact 🟢 Confort · Effort Moyen
> **Statut** : PLAN — aucune modification du SDD Mode

---

## 1. Intention (le POURQUOI)

L'analyse (§2.3 cross-model workflow, §2.4 subagents/orchestration) montre la puissance des **uncorrelated context windows** : un modèle crée un bug, un **autre** modèle le trouve (~80 % des bugs attrapés par une équipe d'agents à l'ouverture de PR). Le pattern cross-model (Claude auteur, Codex/Gemini reviewer) impose une règle d'or : le reviewer **insère des findings sans réécrire** — « additive only ». Transposé à SDD, c'est du **Human Authorship entre IA** : la paternité de l'auteur est préservée, le reviewer ne fait qu'ajouter des observations citées. On ajoute `/sdd validate --cross-model`.

## 2. État actuel SDD Mode

| Élément | Fichier réel |
|---------|--------------|
| Validation | `templates/.claude/sdd/validate.md`, `lib/review.js` (`review`) |
| Multi-runtime | `lib/emit-rules.js` (Cursor, Codex, Gemini déjà connus comme runtimes) |
| Verdicts | `lib/score.js`, (futur) `lib/verdict.js` (§3.4) |

**Manque** : aucune orchestration cross-model ; la review est mono-modèle.

## 3. Cible

`/sdd validate --cross-model <reviewer>` : un modèle tiers (Codex/Gemini) review le diff **contre le vrai codebase** et insère des **Findings additifs** (jamais de réécriture), agrégés avec dedup, sans toucher au verdict déterministe.

## 4. Conception détaillée

### 4.1 Contrat additive-only
- Le reviewer produit un artefact `.aiad/reviews/REVIEW-<spec>-<reviewer>.md` : liste de Findings `{ sévérité, fichier:ligne, description, suggestion }`.
- **Interdit** : modifier le code ou la SPEC (allowlist sans `Edit`/`Write` — même principe que les agents de gouvernance §3.1).
- Les Findings sont **mergés** dans le rapport de validation comme une section additionnelle horodatée et attribuée (« Reviewer: Codex »).

### 4.2 Orchestration
- `lib/cross-model.js` : invoque le runtime tiers (CLI Codex/Gemini, ou subagent avec `model:` différent), passe le diff + le contexte minimal, récupère les Findings (format JSON, schéma `review.schema.json`).
- **Uncorrelated context** : le reviewer reçoit un contexte **frais** (faits vs opinions, §2.4 Dex) — ne pas lui donner le raisonnement de l'auteur.
- **Dedup** : un second passage (ou heuristique `lib/cross-model.js`) déduplique les faux positifs (pattern « agents de dedup parallèles », §2.4).

### 4.3 Verdict
- Le verdict final reste **déterministe** (§3.4) : les Findings cross-model **informent** mais ne remplacent pas le PASS/FAIL/JNSP calculé. Findings de sévérité haute non résolus → peuvent forcer `CONDITIONAL PASS` (§3.6).

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-cross-model-review.
2. Définir `review.schema.json` (Findings) + dossier `.aiad/reviews/`.
3. Implémenter `lib/cross-model.js` (invocation runtime tiers, contexte frais, récupération JSON).
4. Ajouter le flag `--cross-model <reviewer>` à `/sdd validate` (`lib/review.js`).
5. Implémenter le merge additif + dedup.
6. Brancher l'influence sur le verdict (CONDITIONAL si findings hauts).
7. Documenter la disponibilité des runtimes tiers (pré-requis CLI Codex/Gemini installés).

## 6. Fichiers touchés / créés

- **Créés** : `lib/cross-model.js`, `.aiad/schema/verdicts/review.schema.json`, `.aiad/reviews/_index.md`, tests.
- **Modifiés** : `lib/review.js`, `templates/.claude/sdd/validate.md`, `bin/aiad-sdd.js`, doc.

## 7. Critères d'acceptation (EARS)

- **WHEN** `/sdd validate --cross-model` est lancé, **the system SHALL** faire reviewer le diff par un modèle tiers en contexte frais.
- **the system SHALL** restreindre le reviewer à des Findings additifs ; il **SHALL NOT** modifier le code ou la SPEC.
- **the system SHALL** dédupliquer les Findings avant agrégation.
- **the system SHALL** conserver un verdict final déterministe ; les Findings hauts non résolus **SHALL** au plus forcer `CONDITIONAL PASS`.

## 8. Tests & vérification

- Test : reviewer tente une écriture → refusé.
- Test dedup : findings identiques fusionnés.
- Test : findings haute sévérité → verdict CONDITIONAL.
- Test schéma : sortie reviewer conforme à `review.schema.json`.

## 9. Risques, dépendances, rollback

- **Risque** : indisponibilité du runtime tiers (headless/CI) → dégrader proprement (skip + note), jamais bloquer.
- **Risque** : coût (2 modèles) → opt-in explicite.
- **Dépendance** : §3.4 (verdicts), §3.6 (CONDITIONAL PASS), §3.1 (allowlist read-only).
- **Rollback** : flag opt-in, retrait sans impact.

## 10. Effort & découpage en SPECs

- SPEC unique. **Effort Moyen** (l'invocation cross-runtime + le contrat additive sont le cœur).
