# Plan — §3.5 Phase Research + gate GO/NO-GO + Discovery obligatoire

> **Recommandation** : §3.5 de `analyse-claude-code-best-practice.md`
> **Priorité** : P1 · Impact 🟠 Fort · Effort Moyen
> **Statut** : PLAN — aucune modification du SDD Mode
> **Note** : « le plus gros manque » du cycle SDD selon l'analyse.

---

## 1. Intention (le POURQUOI)

Le cycle SDD score aujourd'hui la *qualité de la SPEC* (SQS) mais **jamais la viabilité de l'intention** elle-même. Le mapping RPI↔SDD (§2.3) montre un trou : `RESEARCH.md + GO/NO-GO` n'a **aucun équivalent** SDD. De plus, l'absence de **Discovery codebase obligatoire** avant la spécification est exactement le piège « specs-to-code » dénoncé par Matt/Dex/Karpathy (§2.4, §4) : « le code est ton champ de bataille ». On ajoute donc une phase **Research → GO/NO-GO** entre Intent et SPEC, et un **Discovery ancré dans le code** (agent `Explore` read-only) avant `/sdd spec` et `/sdd exec`. La paternité reste humaine : la Research informe, l'humain tranche le GO/NO-GO.

## 2. État actuel SDD Mode

| Phase | Commande | Lib |
|-------|----------|-----|
| Intention | `/sdd intent` | `templates/.claude/sdd/intent.md` |
| Spécification | `/sdd spec` | `templates/.claude/sdd/spec.md`, `lib/spec-suggester.js` |
| Gate SQS | `/sdd gate` | `lib/score.js` |
| Exécution | `/sdd exec` | `templates/.claude/sdd/exec.md` |

**Manque** : aucune commande entre `intent` et `spec` ; aucun Discovery imposé.

## 3. Cible

1. **`/sdd research`** (nouvelle sous-commande du routeur `/sdd`) : produit un verdict gradué **`GO | CONDITIONAL GO | DEFER | NO-GO` + confidence (%)**, ancré dans le code via Discovery.
2. **Discovery obligatoire** : un agent `Explore` (haiku, read-only) cartographie le code pertinent **avant** `/sdd spec` et `/sdd exec`, et son résultat est un prérequis machine-vérifiable.

## 4. Conception détaillée

### 4.1 Artefact Research
Nouveau dossier `.aiad/research/RESEARCH-NNN-<slug>.md` (parallèle à `intents/`, `specs/`), template `templates/.aiad/research-template.md` :
```md
# RESEARCH-NNN — <titre>  (← INTENT-NNN)
## Discovery (ancrage code — agent Explore)
- Fichiers/zones impactés : …(evidence: chemin:ligne)
- Contraintes existantes : …
## Faisabilité
## Risques & inconnues  → liste de JNSP éventuels
## Verdict : GO | CONDITIONAL GO | DEFER | NO-GO  (confidence: NN %)
## Conditions (si CONDITIONAL GO)
```

### 4.2 Commande `/sdd research`
- Corps : `templates/.claude/sdd/research.md` (dual-mode `--guided`/`--fast`).
- Orchestration interne (inspirée de `/rpi:research`, §2.3) : Discovery (`Explore`) → analyse faisabilité → verdict.
- **`context: fork` + agent `Explore`** pour le Discovery (read-only, haiku, hors contexte principal — garbage collection du contexte).
- Verdict produit par un calcul CLI déterministe `aiad-sdd research-score` (heuristiques : nb d'inconnues, couverture Discovery, JNSP ouverts) → exit 0 (GO), 1 (NO-GO/DEFER), 2 (inconnues = JNSP). Recoupe §3.4.

### 4.3 Discovery obligatoire (gate de prérequis)
- `/sdd spec` et `/sdd exec` vérifient l'existence d'un `RESEARCH-*.md` lié avec Discovery rempli ; sinon **JNSP** : « lance d'abord `/sdd research` ».
- Mécanisme enforced (lien avec P0) : hook `UserPromptExpansion` sur l'expansion de `/sdd spec` qui injecte un `additionalContext` rappelant le prérequis, ou bloque si absent.

### 4.4 Mise à jour du cycle documenté
Le cycle passe de :
`Intent → SPEC → Gate → Exec → Validate → Drift Lock`
à :
`Intent → **Research (GO/NO-GO)** → SPEC → Gate → Exec → Validate → Drift Lock`.
Mise à jour de `CLAUDE.md`, `AGENTS.md`, `SDDMode.md`, `GUIDE.md` via `emit-rules`.

## 5. Étapes d'implémentation séquencées

1. **Spec** : SPEC-NNN-research-phase (+ SPEC-NNN-discovery-gate).
2. Créer le template d'artefact + dossier `.aiad/research/` (+ `_index.md`).
3. Créer `templates/.claude/sdd/research.md` (dual-mode) et l'enregistrer dans le routeur `templates/.claude/commands/sdd.md`.
4. Implémenter `lib/research.js` (score de viabilité + verdict gradué + exit codes via `lib/verdict.js`).
5. Brancher le Discovery `Explore` (`context: fork`, agent `Explore`).
6. Ajouter le prérequis Discovery dans `sdd/spec.md` et `sdd/exec.md` (+ hook `UserPromptExpansion` optionnel).
7. Mettre à jour le cycle dans toute la doc via `emit-rules`.
8. Mettre à jour `/aiad-help` et le tableau des commandes.

- **Modifiés** : `templates/.claude/commands/sdd.md` (routeur), `templates/.claude/sdd/{spec,exec}.md`, `bin/aiad-sdd.js`, `CLAUDE.md`/`AGENTS.md`/`SDDMode.md` (cycle), `lib/emit-rules.js`, `/aiad-help`.

## 7. Critères d'acceptation (EARS)

- **WHEN** un Intent est capturé, **the system SHALL** permettre `/sdd research` produisant un verdict ∈ {GO, CONDITIONAL GO, DEFER, NO-GO} + confidence.
- **the system SHALL** ancrer chaque Research dans le code via un Discovery `Explore` read-only citant des fichiers réels.
- **IF** aucun `RESEARCH-*.md` avec Discovery valide n'est lié, **the system SHALL** refuser `/sdd spec` (verdict JNSP).
- **WHERE** le verdict est `NO-GO` ou `DEFER`, **the system SHALL** ne pas permettre de passer en SPEC sans nouvelle Research.

## 8. Tests & vérification

- Test `lib/research.js` : jeux d'inputs → chaque verdict.
- Test prérequis : `/sdd spec` sans Research → JNSP.
- Test Discovery : l'agent `Explore` ne peut pas écrire (read-only).
- Test doc : le cycle est cohérent dans tous les runtimes émis (CI `emit-rules-check`).

## 9. Risques, dépendances, rollback

- **Risque** : alourdir le cycle (contre « léger par défaut » §4) → rendre `/sdd research` **proportionné** : mode `--fast` ultra-court pour intentions simples ; obligatoire seulement quand l'ambiguïté/risque est élevé (heuristique sur l'Intent).
- **Dépendance** : §3.4 (verdict/exit codes), agent `Explore` (built-in).
- **Rollback** : la commande peut rester optionnelle (warning au lieu de JNSP) via `config.yml`.

## 10. Effort & découpage en SPECs

- SPEC-A : commande `/sdd research` + `lib/research.js` + artefact. *(Moyen)*
- SPEC-B : Discovery `Explore` obligatoire + prérequis sur `spec`/`exec`. *(Moyen)*

**Effort global** : Moyen. Plus forte valeur P1 (comble le trou #1 + meilleure défense anti « specs-to-code »).
