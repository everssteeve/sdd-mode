Cycle step 6 · /sdd validate

Validation — three axes, one verdict

Code that compiles is not necessarily code that works, and code that works is not necessarily code that honours the intent. SDD validation is threefold — technical, functional, governance — and the code is never finished until the SPEC is synchronized (Drift Lock).

Axis 1 — Technical validation

The objective baseline: the project's toolchain, run in full and reported as a table. No judgement, just facts.

Technical validation checks
CheckCommandResult
Lint[project command]PASS / FAIL
Types[type-check]PASS / FAIL
Unit tests[test]PASS / FAIL
Integration tests[test:integration]PASS / FAIL
Build[build]PASS / FAIL
Coverage[coverage]X%
Two absolute prohibitions

NEVER validate code that fails the tests. NEVER validate code containing unresolved TODO-JNSP: markers (JNSP: Je Ne Sais Pas — "I don't know") — these markers are questions awaiting a human decision, not cosmetic details.

Axis 2 — Functional validation

The functional axis confronts the code with the SPEC's test cases, criterion by criterion — against the code itself, not only against the tests. Specified inputs respected, processing steps followed, output format produced, every edge case covered by a test, every acceptance criterion checked off. In addition, the parent Intent's Drift Criterion is re-read: is the observable signal of intent betrayal absent?

Axis 3 — Governance validation

The four Tier 1 agents — AIAD-AI-ACT, AIAD-RGPD, AIAD-RGAA, AIAD-RGESN — are applied via the regulatory-veto skill. Each holds a right of veto: a VETO blocks, a WARN requires a remediation plan before merge. This validation is not optional for the projects concerned.

UNKNOWN = VETO (fail-closed)

When a governance agent cannot decide — a standard not applicable with certainty, missing information — it returns UNKNOWN. That verdict is treated as a VETO: you do not merge on a regulatory doubt. This is the governance flavour of the JNSP policy.

A composition of skills, with no duplication

/sdd validate is the most composite command of the cycle: it implements no logic of its own but composes three reusable skills — ears-validator (coverage of §3's testable criteria), drift-detection (code ↔ SPEC gaps on the diff) and regulatory-veto (the 4 Tier 1 standards). Each skill is also invoked elsewhere in the cycle (/sdd gate, /sdd drift-check, /sdd security…): the logic lives in one place, and the commands orchestrate it.

The verdict

The four validation outcomes
VerdictMeaningAction
VALIDATEDTriple validation passedProceed to the Drift Lock (/sdd drift-check)
FIXESMinor, localized gapsList them, relaunch the agent on the precise points
REJECTEDStructural failureImprecise SPEC → Human Learning · agent wrong despite a clear SPEC → Lesson Learned
JNSPValidation undecidableQuestion(s) for the human — never downgrade to "minor fixes" to push the merge through

The Human Learning / Lesson Learned distinction is essential for continuous improvement: the former documents a gap between human intent and delivery (the SPEC was the problem), the latter a recurring agent error (the execution was the problem). Both feed the AGENT-GUIDE.

What is validated: <PASS axes>
What blocks: <undecidable areas with § ref>
Question(s) for the human: <actionable rephrasing>

The complementary roles — /sdd security and /sdd audit

Two audit commands extend validation for high-stakes features, with reports persisted in .aiad/metrics/:

Complementary audit commands
CommandScopePersistence
/sdd securityOWASP Top 10, secrets management, agent permissions (minimal necessary permissions), data exposure, AI-ACT/RGPD compliance.aiad/metrics/security/YYYY-MM-DD-SPEC-NNN.md
/sdd auditCode ↔ SPEC conformity, introduced technical debt (complexity, coupling, readability), consistency with the AGENT-GUIDE.aiad/metrics/audit/YYYY-MM-DD-SPEC-NNN.md
When to run them

/sdd security: after any feature touching access control, user data, secrets or an AI component — recommended before any critical PR, with a frontier model. /sdd audit: for features with high technical stakes, or after several iterations on the same component. The persisted reports feed the metrics and the retrospectives.
